# 기술인 주소록 — 결정 필요 사항

구현 전 사용자 확인이 필요한 항목. 결정되면 이 문서에 결과를 기록한다.

## ① 기존 직원 테이블과의 관계

엑셀 667명은 외부 기술인력 풀이라 팀 내부용 `overtime_employees`(11명)와 합치지 않고
**별도 `engineer_contacts` 테이블**로 간다. 내부 직원 연결용 `employee_id` 컬럼만
예약(MVP 미사용). 합치면 연장근무·휴가관리 직원 목록에 667명이 섞이는 문제가 있다.

**제안: 별도 테이블 + 연결 컬럼 예약.** → 결정: 승인 (2026-07-15)

## ② 직위/직책 구분

엑셀 원본은 직위 하나만 관리한다(16종). MVP는 **직위 단일 필수 필드**로 시작하고,
직책(팀장·본부장 등)은 선택 입력 필드로만 둔다.

**제안: 직위 필수 + 직책 선택.** → 결정: 승인 (2026-07-15)

## ③ 직위 정렬 순서

관리자 설정 테이블 대신 **코드 상수 배열**(사원→…→총괄사장→고문, 실데이터 16종 반영)로
시작. 순서 변경 요구가 생기면 설정 테이블로 승격.

**제안: 코드 상수.** → 결정: 승인 (2026-07-15)

## ④ 주소 입력 방식

**안 1(단일 텍스트 필드)** + 지역(시·도)만 별도 드롭다운. 엑셀 원본이 단일 주소 문자열이라
분리 입력(안 2)은 이관도 입력도 번거롭다. 우편번호·상세주소 분리와 주소 검색 API는 확장.

**제안: 안 1 + 지역 필드.** → 결정: 승인 (2026-07-15)

## ⑤ 엑셀 초기 데이터 이관

667건이므로 범용 가져오기 UI 대신 **구현 시점에 xls를 파싱해 1회 시드로 직접 이관**
(seed SQL 보관). 지역은 주소에서 best-effort 추출, 직위 이상치 "8" 1건은 빈 값+비고 처리,
중복 전화 3건·동명이인 4명은 실데이터 그대로 이관. 범용 가져오기/내보내기 UI는 확장.

**제안: 1회 직접 시드.** → 결정: 승인 (2026-07-15)

## ⑥ 삭제 방식

`employment_status='비활성'`이 소프트 삭제 역할(기본 목록에서 숨김). 완전 삭제는 상세
모달에서만, 확인창 후 hard delete(참조가 전문분야 junction뿐이라 CASCADE로 안전).
별도 `deleted_at` 컬럼은 두지 않는다.

**제안: 비활성화 기본 + 완전 삭제는 상세에서만.** → 결정: 승인 (2026-07-15)

## ⑦ 전문분야 MVP 포함

다중 지정 구조(마스터+junction) 포함, 12종 시드. 단 이관 데이터에는 분야 정보가 없어
**전원 미지정으로 시작** — 필요할 때 수정 모달에서 채워 넣는 방식.

**제안: 구조·UI 포함, 값은 비워서 시작.** → 결정: 승인 (2026-07-15)

## ⑧ 상세 보기 형태

우측 상시 패널 대신 **상세 모달**(행 클릭) — 이 앱의 기존 상세 보기 패턴(프로젝트 List
툴팁 모달)과 일치하고, 667행 테이블의 가로 폭을 확보한다.

**제안: 모달.** → 결정: 승인 (2026-07-15)

## ⑨ 개인정보 표시·권한

내부 관리자용 전제로 전체 표시. 접근 제어는 방금 도입한 **항목별 권한**을 그대로 사용 —
`engineers` 키 추가로 사용자별 쓰기/읽기/숨김 설정 가능(읽기면 편집 UI 숨김, 복사·검색
허용). 마스킹·내보내기 권한 세분화는 확장.

**제안: 전체 표시 + 항목별 권한 연동.** → 결정: 승인 (2026-07-15)

## ⑩ 카드 보기

MVP 제외(확장). 667명 규모에선 테이블 검색성이 우선이고, 요구 16도 "범위를 과도하게
늘리면 분리" 허용.

**제안: 확장으로 분리.** → 결정: 승인 (2026-07-15)

## 승인 시 추가 요청 (2026-07-15, 전부 반영)

- **④ 보완**: 지역(시·도)은 별도 드롭다운 입력이 아니라 **주소에서 자동 추출**을 기본으로
  한다(`lib/engineers/format.ts`의 `extractRegion` — 저장 시 자동 계산, 수정 모달에서
  덮어쓰기 가능).
- **⑦ 보완**: 전문분야 **열과 필터는 지정 데이터가 하나도 없는 동안 자동으로 숨긴다**
  (junction 행 존재 여부로 판단). 데이터가 생기면 별도 설정 없이 바로 나타난다.
- **③ 보완**: 목록에 없는 새 직위도 오류 없이 표시(정렬만 뒤로). 직위 이상치 "8" 1건은
  임의 변경하지 않고 원본 확인 후 별도 보고.
- **④(기존 데이터)**: 전화 중복·동명이인 자동 병합/삭제 금지 — 별도 인물로 그대로 이관.
- **⑥~⑬(향후 엑셀 동기화 대비)**: 아래 구조를 지금 반영한다 — 상세는 03 문서의
  "향후 엑셀 동기화 설계" 절.
  - `engineer_no` (자동 증가 고유번호) 컬럼 추가 — 내보내기/재가져오기 시 동일인 매칭 키.
  - `engineer_sync_logs` 테이블을 미리 생성 (실행일시·파일명·신규/수정/비활성/오류 건수).
  - 동일인 판정 우선순위: engineer_no → 휴대전화(하이픈 제거) → 성명+직위+주소.
    불확실하면 자동 병합 없이 "확인 필요"로 분류.
  - 사라진 기술인은 자동 처리 금지 — "비활성 후보"로만 표시하고 사용자가
    유지/비활성/퇴사 중 선택.
  - 다른 기능이 기술인을 참조하게 되면 FK는 RESTRICT/SET NULL — 주소록 삭제·비활성이
    이력을 지우지 않게 한다.

## ⑪ 2021년도 전직원 인사기록부 보완 (2026-07-15 1회 실행, 완료)

`recording.xls`(재직자 542명 「(주)선종합」 시트 + 퇴사자 1,478명 「퇴사자명부」 시트, 성명은
글자 사이 공백 포함 원본 — 정규화 후 비교. 「고령자명부」는 전화·주소·직위 필드가 없어 제외,
「중국현지직원」은 급여·신분증 등 민감정보라 범위 밖으로 제외)로 기술인 주소록(667명)의
빈 필드만 1회 보완했다. 주민등록번호 등 민감 인사정보는 어떤 필드로도 저장하지 않았다.

- **대상**: "현장 현황 또는 담당자 정보에 이미 존재하는 사람"(OR 조건) 중 후자를 택해
  **기술인 주소록 전체(667명)**를 대상으로 함 — 현장 담당자 텍스트에서 이름을 다시 파싱하는
  것보다 안전하고, 이미 구조화된 인물 레코드가 있어야 "빈 필드 보완"이 의미가 있기 때문.
- **매칭**: 정규화 성명(공백 전부 제거) + 휴대전화 일치 = 확정 매칭만 자동 반영. 직위·주소
  일치 또는 전화 한쪽 누락 = 높은 가능성(미반영), 그 외 = 판단 불가(미반영). 신규 인물은
  생성하지 않음.
- **열 매핑**: 인사기록부 "직책" 열의 실제 값(상무/부장/사원 등)이 주소록 `rank`(직위)와
  동일 체계임을 확인 후 매핑. "직종"(건축/토목/기계/전기/통신/소방/조경 등)은 주소록의
  전문분야 마스터와 겹치는 값만 매핑, 겹치지 않는 값(사무/환경/구조/연구)은 사용하지 않음.
- **결과**: 확정 매칭 291명(높은 가능성 2 / 판단 불가 67 / 매칭 없음 307, 무시). 빈 필드만
  채움 — 소속 291건, 입사일 291건, 퇴사일+재직상태 12건, 전문분야 태그 282건. 값이 달라
  자동 반영하지 않은 충돌 390건(주소 표기 차이 284, 직위 표기 차이 105, 전화 1 — 형식 차이로
  추정, 어느 것도 덮어쓰지 않음).
- **개인정보 처리**: 원본 파일은 사용자 데스크톱에서 직접 읽었고 저장소에 복사·커밋하지
  않음. 파싱 결과 전체(JSON/CSV/SQL)도 저장소에 남기지 않았고, 세션 스크래치패드에 만들었던
  집계 파일도 처리 완료 후 전부 삭제함. 충돌 목록은 마스킹된 성명 기준으로만 잠시 확인 후 폐기.
