# 기술인 주소록 — 검색·필터·정렬·검증

## 통합 검색

- 페이지 로드 때 667건 + 전문분야 junction을 **한 번에 불러와 클라이언트에서 필터링**한다
  — projects/휴가관리와 같은 패턴. 이 규모(수백 건)에선 서버 검색이 오히려 느리고 복잡하다.
  수천 건 규모가 되면 그때 Supabase `ilike`/FTS로 전환.
- 검색 대상: 성명, 직위, 직책, 소속, 전문분야명, 핸드폰, 사무실 전화, 이메일, 지역, 주소, 비고.
- **debounce 250ms**: 입력값은 즉시 state에 반영하되 필터 계산에 쓰는 검색어만 250ms 지연
  (`useEffect` + `setTimeout`). 667건이라 사실 없어도 되지만 요구 명세대로.
- 전화번호 검색은 **하이픈·공백 제거 후 비교** — "01012345678"로 쳐도 "010-1234-5678"이
  잡힌다. 그 외 필드는 소문자 포함(includes) 매칭.
- 결과 없음: "검색 조건에 맞는 기술인이 없습니다."

## 필터 (동시 적용, AND 결합)

| 필터 | 옵션 | 비고 |
|---|---|---|
| 재직 상태 | 전체/재직/퇴직/비활성 | 기본 **재직** (비활성 기본 숨김 — 요구 13) |
| 직위 | 전체 + 데이터에 존재하는 직위 목록 | 데이터에서 동적 생성 |
| 전문분야 | 전체 + 마스터 목록 | junction 조인 결과로 매칭 |
| 지역 | 전체 + 데이터에 존재하는 시·도 | `region` 필드 기준 |
| 즐겨찾기만 | 토글 | |

[필터 초기화] 버튼 = 검색어 포함 전부 리셋.

## 정렬 (드롭다운 단일 선택)

기본: **재직 우선 → 즐겨찾기 우선 → 성명 가나다순** (요구 8의 "재직자 우선 + 즐겨찾기
상단"을 기본값에 내장).

옵션: 성명 가나다순(기본) / 직위순 / 소속순 / 입사일순 / 최근 수정순.
직위순은 가나다가 아니라 **코드 상수의 직위 서열**(03 문서 §4) 기준. 어떤 정렬을 골라도
재직·즐겨찾기 우선 규칙은 앞단에 유지된다.

## 전화번호 자동 포맷

숫자만 입력해도 하이픈 자동 삽입, 표시 형식 그대로 저장·복사(요구 9):

- `010########` → 010-####-#### (11자리)
- `02#######~########` → 02-###-#### / 02-####-#### (서울 2자리 국번)
- 그 외 3자리 지역번호(031/043/…): 0##-###-#### / 0##-####-####
- 규칙에 안 맞는 길이는 입력값 그대로 두고 저장 허용(내선·해외 번호 대비, 형식 오류는
  휴대폰 패턴일 때만 경고).

`lib/engineers/format.ts`의 순수 함수 `formatPhone()`으로 구현, 추가/수정 모달의
onChange에서 적용.

## 입력값 검증 (저장 전)

| 검사 | 처리 |
|---|---|
| 성명/직위/핸드폰/주소 미입력 | 차단 |
| 이메일 형식 오류 (`@` 포함 간단 패턴) | 차단 |
| 퇴사일 < 입사일 | 차단 |
| 재직 상태인데 퇴사일 입력 | 경고 (저장 허용 — 예정 퇴사 입력 가능) |
| 퇴직 상태인데 퇴사일 없음 | 경고 (저장 허용) |
| **중복 전화** (하이픈·공백 제거 비교, 자기 자신 제외) | **경고 후 confirm 저장 허용** — 실데이터에 이미 중복 3건 존재 |
| 중복 이메일 | 경고 후 허용 |
| 동일 성명+동일 전화 | 경고 후 허용 (완전 중복 의심) |

경고 문구 예: "동일한 전화번호를 사용하는 기술인이 이미 등록되어 있습니다: 홍길동. 그래도 저장하시겠습니까?"
