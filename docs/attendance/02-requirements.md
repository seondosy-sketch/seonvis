# 기술인 출퇴근부 — 요구사항 정리

> 마스터 프롬프트(사용자 지시)에서 **명시적으로 확정한 사항**은 Confirmed로,
> 마스터 프롬프트가 "여러 방안 중 선택해라/검토해라"라고 위임한 부분은 분석자의 권장안을 Proposed로,
> 마스터 프롬프트로도 결정할 수 없는 사항은 Open Question으로 분류한다.
> 근거 코드/파일은 `01-current-analysis.md` 참고.

---

## 1. 기간 기준

**Confirmed**
- 1개월 = 전월 21일 ~ 당월 20일 (연장근무와 동일 규칙, `lib/overtime/summary.ts` 재사용).
- 연간 마감 기준 = 해당 연도 12월 20일. 1년 통합 범위 = 전년도 12월 21일 ~ 해당 연도 12월 20일.
- 기간 계산은 공통 유틸/회계기간 설정으로 관리(코드 곳곳에 하드코딩 금지). 연장근무 로직 재사용, 중복 작성 금지.

**Open Question**
- 첨부 엑셀 실물은 연말 블록을 12/20이 아니라 12/31까지 그려둔다(`01-current-analysis.md` §7.2). 연간 통합 출력의 실제 종료일을 마스터 프롬프트 규칙대로 **12/20으로 엄격히 자를지**, 아니면 기존 엑셀 습관대로 **12/31까지 표시하되 12/21~31 구간은 "다음 연도 1월분 마감 전 임시 데이터"로 표시할지** 확인 필요.

## 2. Project List 연계

**Confirmed**
- 프로젝트·기술인을 출퇴근부에 별도 중복 입력하지 않고 Project List 기준으로 연계.
- 연계 대상(프로젝트): ID, 프로젝트명, 공고일, 면접일, 공고 상태, 취소 여부, 재공고 여부, 변경공고 여부, 진행상태.
- 연계 대상(참여기술인): 기술인 ID, 성명, 참여 직책, 분야, 참여 시작일/종료일, 참여 상태, 단장 여부.
- 참여기술인이 변경돼도 과거 출근기록은 삭제/덮어쓰기 금지. 기록 생성 시점의 프로젝트명·기술인 성명·직책·분야·참여기간 스냅샷 보존 여부 검토.
- Project List 수정 후에도 이미 마감된 월별 명부·과거 출력물은 동일하게 재현돼야 함.

**Proposed** (→ 상세 근거는 `03-data-model.md`)
- `projects`에는 재공고/변경공고 여부 컬럼이 없다(01문서 §2.2) — 신규 컬럼 2개(`is_reannounced`, `is_amended` 또는 이력 테이블)를 `projects`에 추가하는 안을 제안. Project List 화면에도 이 상태를 노출해야 Project List 쪽 사용자에게도 의미가 있다(출퇴근부 전용 필드로 숨기면 Project List와 정보가 어긋남).
- "참여기술인" 연계는 **신규 관계 테이블**(`project_participants` 가칭)로 구현 — `engineer_contacts`를 기술인 마스터로, `projects`를 프로젝트 마스터로 참조하는 junction + 참여기간/직책/단장여부/참여상태 컬럼 보유. 기존 `director`/`staff_*` 텍스트 필드는 그대로 두고 건드리지 않는다(Project List의 다른 화면·주간보고 로직이 이 필드들을 이미 참조 중이므로 — `docs/features.md` §2, §3).
- 과거 기록 보존 방식은 "월 마감 시 스냅샷"으로 제안(→ `03-data-model.md` §의 결정 사유 참고). 참여기술인 이력 테이블을 별도로 두는 대신, `project_participants`에 유효기간(참여시작일/종료일 + 참여상태)을 두어 "현재 유효한 연계"만 걸러 보여주고, 월 마감 시점에는 그 시점 유효했던 참여자 스냅샷을 별도 저장한다.

**Open Question**
- "재공고 여부"/"변경공고 여부"를 `projects`에 진짜 boolean 컬럼으로 추가할지, 아니면 프로젝트 변경이력 테이블(6번 항목)의 최신 이벤트로 파생시킬지 — 두 방식 다 가능하고 트레이드오프가 있어 사용자 결정 필요.

## 3. 출퇴근부 적용기간 (기술인별 체크 가능 기간)

**Confirmed**
- 기본기간 = 공고일 ~ 면접일. 그 밖 날짜는 기본적으로 체크 비활성화.
- 관리자가 기본기간과 별개로 "출근부 적용기간"을 예외 조정할 수 있어야 함(사전 준비업무, 후속업무, 면접일 변경, 재공고, 변경공고, 공고 취소 후 재개, 일정 미확정 대응).
- 면접일 미입력 프로젝트의 처리 규칙은 임의로 정하지 말고 Project List 데이터 구조·운영방식 확인 후 제안.

**Confirmed (01문서 근거)**
- `projects.interview_date`는 날짜 형식이 아닐 수 있다("서면", "추후" 같은 자유 텍스트 — `01-current-analysis.md` §2.1, `lib/projectStatus.ts`의 `categorizeProject`가 이미 이 경우들을 분기 처리 중). 즉 "면접일 미입력"뿐 아니라 "면접일이 날짜가 아닌 텍스트"인 경우도 반드시 같은 방식으로 처리해야 한다.

**Proposed**
- 프로젝트별로 `attendance_period_start`/`attendance_period_end`(nullable, date) 컬럼을 별도로 두어 "기본기간(공고일~면접일)"과 "실제 적용기간"을 분리 — null이면 기본기간 그대로 사용, 값이 있으면 그 값으로 덮어씀(연장근무의 `overtime_projects.start_date/end_date`가 이미 이 패턴을 씀 — 01문서 §8 재사용 목록 참고 가능).
- 면접일이 없거나("추후") 텍스트("서면")인 프로젝트는 **종료일 없이 계속 체크 가능**(연장근무가 이미 "발표일 없으면 end_date null = 종료일 없이 계속 표기"로 동일한 문제를 해결한 선례 — `docs/overtime.md` "입찰 프로젝트 자동 연계" 섹션)로 통일. 화면에는 "면접일 미확정 — 관리자 확인 필요" 경고 배지를 표시.

**Open Question**
- 위 Proposed(연장근무와 동일 처리)로 확정할지, 아니면 "현재 월 기준일까지만 임시 허용" 등 다른 방식을 원하는지 확인 필요.

## 4. 월별 출퇴근부 화면

**Confirmed** — 상단 기능
기준연월 선택 · 프로젝트 검색 · 프로젝트 상태 필터 · 분야 필터 · 기술인 검색 · 출근기록 저장 상태 · 월 마감 상태 · 월 출력 · 월 마감 · 마감취소/재개(권한자만).

**Confirmed** — 행 구조
프로젝트명 · 직책 · 분야 · 성명 · 날짜별 체크 · 출근일수 · 비고. 날짜 열은 선택 월의 전월 21일~당월 20일 자동 생성. 날짜 헤더에 일자+요일 최소 표시, 토·일·공휴일 시각 구분.

**Confirmed (01문서 근거로 보강)**
- 요일 표시는 첨부 엑셀에는 **없었다**(일자 숫자만, 01문서 §7.4) — 이번 화면은 마스터 프롬프트 지시대로 요일을 새로 추가하는 것이며 엑셀 재현이 아니다.
- 주말/공휴일 표시도 엑셀에는 사실상 없었으므로(§7.4), `holidays`/`/api/holidays`를 이용해 **새로 계산해 구현**해야 한다.

## 5. 출근 체크 방식

**Confirmed**
- 셀 클릭/체크로 출근 여부 입력. 미출근=기본배경, 출근=음영. 새로고침 후에도 유지.
- 즉시저장 vs 명시적 저장 중 기존 미래Hub UX와 일치하는 방식 적용.
- 다중 셀 선택/드래그, 행 전체 일괄 체크, 선택 날짜 전체 일괄 체크 검토.
- 잘못 입력한 체크 해제 가능. 마감된 월은 수정 불가(마감취소 권한자만 재개).
- DB 저장 항목: 프로젝트 ID, 기술인 ID, 출근일자, 출근 여부, 입력자, 생성일시, 수정자, 수정일시(+ 필요시 참여기술인 관계 ID, 출근기록 상태, 수정사유, 원본 월마감 ID).
- 동일 프로젝트·기술인·날짜 중복 저장 방지 유니크 제약 검토.

**Confirmed (기존 관례 근거)**
- "즉시저장"이 이 코드베이스의 기존 관례다 — 연장근무의 직원/프로젝트 관리 모달, project_notes 등 전부 `onBlur`/`onChange` 즉시 저장이고 별도 "저장" 버튼이 없다(`docs/overtime.md` 6단계, `docs/conventions.md`). 출근 체크도 클릭 즉시 저장 + upsert가 이 프로젝트의 일관된 UX.
- 유니크 제약은 `(project_id, engineer_id 또는 participant_id, work_date)`로 구성 — 연장근무의 `overtime_project_members (project_id, employee_id)` UNIQUE, `leave_types` 등 기존 테이블들이 전부 이런 방식의 복합 UNIQUE를 이미 채택 중.

**Proposed**
- 다중 셀 드래그 선택은 1차 구현에서 제외하고, "행 전체 일괄 체크"(평일만/전체 선택/전체 해제)와 "선택 날짜 전체 인원 일괄 체크"만 먼저 구현 — 연장근무의 `BulkWorkRecordModal`이 이미 "체크 pill + 전체선택/평일만/전체해제" UI 패턴을 갖고 있어 그대로 응용 가능(`docs/overtime.md` "그리드 열 너비/일괄 입력" 섹션). 드래그 선택은 구현 난이도 대비 실사용 빈도가 낮아 향후 과제로 분리 제안.

**Open Question**
- 다중 셀 드래그 입력을 1차 범위에 포함할지 여부(위 Proposed 확인 필요).

## 6. 비고 및 변경이력

**Confirmed**
- 비고는 프로젝트 변경사항 추적 가능해야 함(단장교체·기술인교체·공고취소·재공고·변경공고·공고일변경·면접일변경·참여분야변경·기타).
- "월별 출근명부 비고"와 "프로젝트 변경이력"을 구분 검토. Project List에 이미 변경이력이 있으면 연계, 없으면 신규 구조(프로젝트ID·변경유형·변경일자·변경전/후·메모·작성자·작성일시) 검토.
- 출력 시 해당 기간 변경사항을 날짜순으로 비고란에 나열(예시 포맷 제공됨). 줄바꿈+행높이 자동조정 필요.

**Confirmed (01문서 근거)**
- Project List에는 프로젝트 변경이력 테이블이 **없다**(`project_notes`는 필드별 메모 1개씩만 저장하는 구조라 "이력"이 아니라 "현재 메모"임 — `docs/database.md`). 따라서 "신규 구조" 쪽으로 갈 수밖에 없다.
- 첨부 엑셀 실측(01문서 §7.4, §7.5): 실무자는 비고칸(5건)보다 **날짜 셀 안에 직접** 이벤트를 적어 넣는 경우(187건)가 압도적으로 많았다. 즉 "이 변경이 언제 있었는지"를 표현하는 실제 니즈는 강하고, 변경이력에 **발생일자**가 반드시 있어야 한다는 근거가 명확하다.

**Proposed**
- 신규 `project_change_history` 테이블(변경유형/변경일자/변경전·후/메모/작성자/작성일시)을 만들고, 출퇴근부는 이 테이블을 "연계"해서만 보여준다(자체 이력을 별도로 관리하지 않음 — 마스터 프롬프트 지시 "중복 관리하지 말라"와 일치).
- 변경유형은 고정 enum(`director_change`|`participant_change`|`cancelled`|`reannounced`|`amended`|`announce_date_change`|`interview_date_change`|`field_change`|`other`)으로 관리 — Project List 화면에서도 이 이력을 볼 수 있게 만들면 Project List 자체에도 가치가 생긴다(마감 검증 항목 "변경사항 비고 미작성 여부"와도 연결됨).
- 엑셀 실무에서 관찰된 "날짜 셀에 직접 적은 이벤트"(심사/교체/취소/재공고)는 `project_change_history`의 `change_date`가 그 발생일이 되도록 매핑 — 별도의 "날짜 셀 이벤트" 개념을 새로 만들지 않고 변경이력 하나로 통합.

**Open Question**
- 월별 출근명부 비고(그 달 한정 메모)와 프로젝트 변경이력(발생일 기준 전체 이력)을 화면상 완전히 분리된 두 영역으로 보여줄지, 하나의 비고 칼럼에 "이번 달에 해당하는 이력만 필터링해" 합쳐 보여줄지 확인 필요.

## 7. 월 마감

**Confirmed**
- 19일부터 마감 가능 안내, 20일까지 미마감 시 경고. 시스템이 날짜만으로 강제 마감하지 않고 사용자가 명시적으로 실행.
- 마감 전 검증(오류 vs 경고 구분): 오류 = 중복 출근기록, 필수 식별정보 누락 / 경고 = 면접일 미입력, 비고 미작성 등.
- 마감 시 프로젝트·참여기술인·비고 데이터 스냅샷 저장. 마감자·마감일시 기록. 마감 후 수정 잠금.
- 마감취소는 권한자만, 사유 필수 입력, 이력 보존.

**Confirmed (01문서 근거)**
- "월 마감" 전용 권한 레벨이 현재 권한 체계(`menu_permissions` 3단계)에 없다(01문서 §6) — 신규 권한 개념이 반드시 필요.

**Open Question**
- 마감/마감취소 권한을 어떻게 표현할지: (a) `ADMIN_EMAILS` 관리자만 가능하게 제한, (b) `menu_permissions`에 이 메뉴 전용 4번째 값을 추가(예: `write_close`), (c) 별도 권한 컬럼(`allowed_users.can_close_attendance boolean`) 신설. 세 방식 모두 구현 가능하며 트레이드오프가 다르다 → 사용자 선택 필요(→ `03-data-model.md`에 3안 비교 정리).

## 8. 월별 출력

**Confirmed**
- 제목·대상기간·프로젝트명·직책·분야·성명·날짜별 음영·출근일수·비고·출력일·마감여부 포함. XLSX 최소 지원, PDF/인쇄 미리보기 추가 검토.
- 인쇄: 가로방향, 반복 열/제목행, 프로젝트당 참여기술인 행이 페이지 경계에서 불필요하게 분리되지 않게, 비고 줄바꿈, 여백/페이지번호.
- 출근 셀은 값보다 음영 위주지만 접근성을 위해 셀 내부에 `1`/`○`/`출근`/`빈값+배경색` 중 하나를 함께 저장하는 방안 검토.

**Confirmed (01문서 근거)**
- 이 코드베이스에는 xlsx 바이너리 생성 라이브러리가 없다(01문서 §4.1) — 신규 의존성 도입이 전제조건.
- 첨부 엑셀 자체에도 인쇄영역/반복행/배율 등 인쇄 설정이 전혀 저장돼 있지 않았다(01문서 §7.6) — "기존 엑셀과 유사하게"는 열 구성·음영 방식·비고 형태에 한정되고, 인�기 설정은 이번에 처음 설계하는 것.

**Proposed → 상세는 `05-export-spec.md`**
- 화면 셀 값도 빈 값 + 배경색으로 유지하되(엑셀 관행과 동일), DB에는 항상 명시적 boolean(`is_present`)을 저장 — "값"과 "화면 표시"를 분리(이미 요구사항에 있음).
