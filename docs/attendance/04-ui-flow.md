# 기술인 출근부 — 화면 구조 및 사용자 흐름

> 기존 컨벤션 그대로 따른다: 전체 inline style(`docs/conventions.md`), 라우트는 `app/(dashboard)/attendance/`,
> 라우트 전용 컴포넌트는 `_components/`, 모바일 분기는 `useIsMobile()`(768px). 연장근무 화면(`MonthGrid`, `ProjectGrid`,
> `OvertimeEntryPopover`)의 구조를 최대한 그대로 응용한다 — 이 기능은 사실상 "연장근무 그리드의 프로젝트별 보기"를
> 출근 체크용으로 변형한 것에 가깝다.

---

## 1. 라우팅 (Proposed)

```
/attendance              — 월별 출근부(기본 화면)
/attendance/annual       — 연간 통합 출근명부(조회·출력 전용)
/attendance/print        — 월별 출력 미리보기(연장근무 /overtime/print와 동일 패턴)
```
사이드바: `app/sidebar.tsx`의 GROUPS에 항목 추가(연장근무처럼 근태관리 하위 메뉴로 배치할지, 최상위로 둘지는 사용자 확인 — 근태관리 그룹이 이미 있으므로 그 하위가 자연스러움).

## 2. 월별 출근부 화면 (`/attendance`)

### 2.1 상단 바 (마스터 프롬프트 4번 항목 — Confirmed 항목 그대로, 배치만 Proposed)

```
[‹ 2026년 8월분 (7/21~8/20) ›]  [프로젝트 검색] [상태▾] [분야▾] [기술인 검색]
                                                            [저장됨 ✓] [마감상태: 미마감]
[월 마감]  [마감취소]*  [엑셀 출력]  [인쇄]
* 마감취소 버튼은 can_close_attendance 권한자 + 이미 마감된 달에서만 노출
```
- 기준연월 이동은 연장근무와 동일한 `‹ ›` + "(7/21~8/20)" 괄호 병기 방식(`docs/overtime.md` "한 달의 기준" 섹션 그대로 재사용) — 새 UI 패턴을 만들지 않는다.
- "출근기록 저장 상태"는 즉시저장 방식(§4 참고)이므로 "저장 중.../저장됨" 토스트 형태로 충분 — 별도 "저장" 버튼 불필요.
- "월 마감 상태" 배지: 미마감(회색) / 마감완료(초록, 마감자·마감일시 hover 툴팁) / 마감취소됨(주황, 최근 취소 사유 hover 툴팁).

### 2.2 그리드 (마스터 프롬프트 4번 — Confirmed 행 구조)

연장근무 `ProjectGrid.tsx`(프로젝트 × 담당직원, 프로젝트명 rowSpan 병합, 날짜 열, 기간 밖 음영 처리)의 구조를 그대로 응용:

```
| 프로젝트명(고정) | 직책(고정) | 분야(고정) | 성명(고정) | 7/21 | 7/22 | … | 8/20 | 출근일수 | 비고 |
| ○○현장          | 단장       | 건축       | (기술인A)  |  ■   |      | … |  ■   |   18    | 03.14 단장 교체 |
| (병합, 표시 안 함)| —         | 건축       | (기술인B)  |      |  ■   | … |      |   12    |          |
```
- **고정 열(sticky)**: 프로젝트명/직책/분야/성명 4열 — `MonthGrid`의 첫 컬럼 sticky 패턴을 4열로 확장.
- **고정 헤더**: 날짜 행 — 동일 패턴을 세로 방향에도 적용.
- **프로젝트명 표시 방식(Proposed)**: 첨부 엑셀은 병합이 아니라 첫 행만 값이 있고 나머지는 빈 칸이었다(`01-current-analysis.md` §7.3) — 화면에서는 데이터 정합성과 접근성을 위해 **실제 `rowSpan` 병합 셀**을 사용(연장근무 `ProjectGrid`가 이미 이 방식을 씀). 엑셀 원본의 "빈 칸" 방식을 그대로 재현하지 않는다(마스터 프롬프트 "화면을 그대로 복제하기보다 정상화" 지시와 일치). 단, **Excel 출력물**은 기존 엑셀과 유사해야 하므로 내보낼 때는 병합 셀로 변환(`05-export-spec.md`).
- **날짜 헤더**: 일자 숫자 + 요일(첨부 엑셀에는 없었던 것을 신규 추가, `01-current-analysis.md` §7.4/§4 확인). 토요일 파란 계열, 일요일/공휴일 빨간 계열로 구분(이 코드베이스 기존 팔레트 재사용 — `docs/conventions.md`의 색상 표에서 위험/포인트 색 계열 채용, 새 브랜드 색 도입 안 함).
- **셀 상태**:
  - 빈 배경 = 미출근
  - 음영(회색, 연장근무 팔레트의 중립 색 재사용) = 출근
  - 프로젝트 기간(공고일~적용기간) 밖 = 더 진한 음영 + 클릭 불가(연장근무 `ProjectGrid`의 "기간 안/밖 배경 구분" 패턴 그대로 재사용 — 대각선 빗금은 사용자 피드백으로 이미 기각된 방식이므로 채택하지 않음)
  - 오늘 날짜 열 = 세로 강조선(연장근무에 없는 신규 요소, 마스터 프롬프트 11번 "오늘 날짜 강조" 반영)
  - 마감된 월 = 전체 그리드 읽기 전용(연한 회색 오버레이 + 커서 not-allowed)

### 2.3 셀 클릭 동작 (Proposed)

연장근무의 "셀 팝오버"(`OvertimeEntryPopover`) 대신, 출근 체크는 훨씬 단순(boolean 토글)하므로 **팝오버 없이 클릭 = 즉시 토글**을 기본으로 한다:
- 좌클릭: 출근 ↔ 미출근 토글, `upsert`/`delete` 즉시 실행(§낙관적 업데이트: 실패 시 롤백 + 에러 토스트).
- 마감된 월 또는 프로젝트 기간 밖: 클릭 무시(커서 변화로 사전 안내).
- **일괄 체크(마스터 프롬프트 5번 — Proposed 범위, `02-requirements.md` §5 참고)**:
  - 행 헤더의 "일괄" 아이콘 → 그 기술인의 이번 기간 전체에 대해 "평일만 체크/전체 체크/전체 해제" 모달(연장근무 `BulkWorkRecordModal`의 "전체선택/평일만/전체해제" 패턴 재사용).
  - 날짜 헤더의 "일괄" 아이콘 → 그 날짜에 대해 "표시되는 모든 기술인 체크/해제" (프로젝트 상태·기간 필터가 적용된 현재 화면 기준).
  - 다중 셀 드래그 선택은 1차 범위 제외(Open Question, `02-requirements.md` #4).

### 2.4 비고 열

- 그 달(전월21~당월20)에 해당하는 `project_change_history` 이벤트를 날짜순으로 줄바꿈해 표시(마스터 프롬프트 예시 포맷: `03.14 단장 홍길동 → 김철수 변경`).
- 셀 클릭 → 이력 추가/조회 팝업(변경유형 선택 + 발생일자 + 메모).
- 텍스트 길이에 따라 행 높이 자동 확장(CSS `white-space: pre-wrap` — 엑셀의 wrapText와 동일 효과).

## 3. 연간 통합 명부 화면 (`/attendance/annual`)

마스터 프롬프트 11번 지시대로 **조회·출력 전용**, 편집 UI 없음.

- 상단: 연도 선택, "확정본"/"임시본" 토글(미마감 월 포함 여부).
- 프로젝트 단위로 접기/펼치기(기본 접힘 — 성명 행까지 펼쳐야 상세가 보임).
- 월 경계는 회사 기준월(전월21~당월20)로 세로 구분선 + 상단에 작게 "N월" 라벨(연장근무의 "달력 월이 바뀌는 첫 칸에 작게 M월 표시 + 굵은 좌측 테두리" 패턴 재사용, `docs/overtime.md` "한 달의 기준" 섹션).
- 가로/세로 가상 스크롤 필수(365~376개 날짜 열 — `01-current-analysis.md` §7.7 실측 근거). 프로젝트명/직책/분야/성명 4열은 고정.
- 미마감 월이 하나라도 있으면 상단에 경고 배너("N월이 아직 마감되지 않았습니다 — 임시본으로 표시됩니다") + 그 월의 날짜 열 헤더에 옅은 줄무늬 표시.
- 데이터 소스(Proposed): 이미 마감된 달은 `attendance_closure_snapshot_rows`에서, 아직 마감되지 않은 달(임시본 조회 시)은 `attendance_records`+`project_participants` 라이브 조인에서 가져와 화면에서 이어붙인다 — 이 조립 로직은 연간 화면과 연간 출력이 공유하는 공용 함수로 작성(`lib/attendance/annual.ts` 가칭, §`06-implementation-plan.md`).

## 4. 즉시저장 vs 명시적 저장 (마스터 프롬프트 5번 — Confirmed 방침 확정)

`01-current-analysis.md` §6에서 확인했듯 이 코드베이스의 기존 관례는 전부 즉시저장(`onBlur`/`onChange` 즉시 `upsert`, 별도 저장 버튼 없음)이다. 출근 체크·비고 입력 전부 이 방식을 따른다. "출근기록 저장 상태" UI 요구사항은 "저장 버튼 필요"가 아니라 "지금 저장 중/저장 완료/실패"를 알려주는 낙관적 UI 피드백으로 해석한다.

## 5. 권한에 따른 화면 분기 (마스터 프롬프트 12번)

| 권한 | 화면 동작 |
|---|---|
| 조회만(`read` 또는 권한 없음) | 그리드는 보이되 셀 클릭 비활성화, 비고 편집 불가, 마감/출력 버튼 숨김(출력은 조회 권한자도 가능하도록 별도 취급 — 아래 참고) |
| 입력·수정(`write`) | 셀 체크, 비고 작성 가능. 마감/마감취소 버튼은 숨김(권한 없음) |
| 마감 권한(`can_close_attendance`, `03-data-model.md` §8 Proposed) | 위 + 마감/마감취소 버튼 노출 |
| 관리자(`ADMIN_EMAILS`) | 항상 전체 가능(기존 관례) |

- "월별 출력"/"연간 출력"은 마스터 프롬프트가 조회 권한과 별개 항목으로 나열했으나(12번), 이 코드베이스에 "출력 전용 권한"이라는 개념 자체가 없다 — **Proposed**: 출력은 `read` 이상이면 누구나 가능(추가 권한 단계를 만들지 않음, 과잉 설계 방지). 이견 있으면 Open Question으로 별도 처리 필요.

## 6. 모바일 대응 (Confirmed 관례 적용)

- 그리드 자체는 모바일에서 실용성이 낮음(376개 날짜 열) — `useIsMobile()`이 true면 "이 화면은 데스크톱에서 이용해 주세요" 안내 + **오늘 날짜 기준 요약 카드**(내 프로젝트별 이번 달 출근일수만 텍스트로) 정도로 축소 제공(Proposed, 연장근무도 모바일에서 그리드 자체를 축소하기보다 우선순위 낮은 화면으로 취급하는 기존 태도와 일치).
