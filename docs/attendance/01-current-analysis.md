## 1. 코드베이스 구조 (Confirmed)

### 1.1 전체 아키텍처
- Next.js 16 App Router + React 19, **전체 inline style**(Tailwind는 reset 전용) — `docs/conventions.md`.
- Supabase(Postgres + Auth). 클라이언트는 용도별로 분리: `lib/supabase-browser.ts`(Client), `lib/supabase-server.ts`(Server/API), `lib/supabase-admin.ts`(RLS 우회, 서버 전용).
- 인증: Google OAuth → `allowed_users` 테이블에 있어야 접근(`ADMIN_EMAILS`는 테이블 없이도 항상 전체 접근) — `docs/architecture.md`.
- 쿼리 레이어(`queries.ts` 등 공용 repository)는 **존재하지 않는다** — 각 페이지/모달 컴포넌트가 `createSupabaseBrowserClient()`로 직접 `supabase.from(...)` 호출 (`docs/overtime.md` 6단계 결정사항). 이 코드베이스에는 이 패턴 외의 레이어가 없다.
- 라우트는 `app/(dashboard)/<feature>/`, 라우트 전용 컴포넌트는 `_components/`(Next.js private folder), 공용 도메인 타입/유틸은 `lib/<feature>/`.
- 필드명은 DB 컬럼명과 1:1(camelCase 변환 레이어 없음), 날짜는 `YYYY-MM-DD` 텍스트/date, 화면 표시만 상황별 포맷.
- 날짜 파싱은 반드시 `parseLocalDate()`(로컬 타임존 생성자) 사용 — `new Date("YYYY-MM-DD")`는 한국에서 하루 밀리는 UTC 버그가 있음(`docs/conventions.md`, `lib/projectStatus.ts`).

### 1.2 가장 유사한 선행 기능들
이번 작업과 똑같은 절차(마스터 프롬프트 → 문서 작성 → 승인 → 단계별 구현)를 거친 선례가 이미 3건 있고, 문서 스타일·의사결정 기록 방식을 그대로 따르는 것이 이 코드베이스의 관례다.

| 기능 | 문서 | 비고 |
|---|---|---|
| 연장근무 관리 | `docs/overtime.md` | "전월 21일~당월 20일" 기준을 이미 구현·정착시킨 유일한 기능 |
| 휴가관리 | `docs/leave-management/01~06` | `overtime_employees` 공용, 날짜별 전개 테이블(`leave_record_dates`) 패턴 |
| 기술인 주소록 | `docs/engineer-address-book/01~06` | 외부 667명 기술인 풀 — 이번 기능의 "기술인 ID" 후보 |
| 현장 현황 | `docs/site-status/01~07` | 엑셀 분석 문서(`01-excel-analysis.md`)가 이번 문서와 가장 유사한 선례 |

## 2. Project List (`projects` 테이블) 분석 (Confirmed)

파일: `app/(dashboard)/projects/page.tsx`(795줄, 쿼리 레이어 없이 페이지 내부에서 직접 CRUD), 스키마: `docs/database.md`.

### 2.1 프로젝트 필드 (관련 부분만)
| 컬럼 | 설명 |
|---|---|
| `id` (uuid), `project_number`(text, 예: A001) | PK, 사람이 읽는 번호 |
| `name` | 용역명 |
| `announce_date` | 공고일 (YYYY-MM-DD 텍스트) |
| `interview_date` | 발표/면접일 — **날짜가 아니라 "서면"/"추후" 같은 자유 텍스트도 들어갈 수 있음** |
| `director` | 단장 — **텍스트 1명 이름**, 구조화된 참여자 목록이 아님 |
| `staff_arch` / `staff_civil` / `staff_mech` / `staff_safety` | 건축/토목/기계/안전 담당자 — **분야당 텍스트 1명씩**, 배열/관계가 아님 |
| `participants` | 참여사(회사) 목록 텍스트. `"드랍"`/`"드롭"` 포함 시 취소로 간주 |
| `status_override` | 수동 상태 지정("취소" 등) |
| `evaluation` | 낙찰사("선"=자사 수주) |

### 2.2 상태 계산 (`computeProjectStatus`, `lib/projectStatus.ts`)
```
status_override 있음 → 그 값
participants에 "드랍"/"드롭" 포함 → 취소
evaluation === "선" → 수주
result_score 또는 evaluation 비어있음 → 진행중
나머지 → 탈락
```
**"재공고"/"변경공고" 전용 상태 값은 없다.** 오직 위 5가지(취소/수주/진행중/탈락 + status_override 임의값)뿐이다.

### 2.3 핵심 갭 — 참여기술인 구조 데이터 부재 (Confirmed)
`projects` 테이블 **어디에도 "여러 참여기술인 + 직책/분야/참여기간/단장여부"를 구조화해 담는 컬럼·관계 테이블이 없다.** `director`/`staff_*` 5개 텍스트 필드가 전부이며:
- 프로젝트에 참여기술인이 5명을 넘어도(첨부 엑셀 실측: 최대 4명, 평균 2.24명이지만 이는 표본일 뿐 상한 근거는 아님) 담을 곳이 없다.
- 참여기간(시작/종료), 단장여부(단장 교체 이력), 참여상태(진행/중도이탈) 등 이번 기능이 요구하는 정보가 전혀 없다.
- 따라서 "Project List의 참여기술인 정보 연계"는 **기존 컬럼을 읽는 것만으로는 불가능하고, 신규 관계 테이블이 필요하다** (→ `03-data-model.md`).

### 2.4 기술인 마스터 후보 — `engineer_contacts` (Confirmed)
`docs/engineer-address-book/03-data-model.md`, `docs/database.md` 기준:
- 외부 기술인력 풀 667명. 컬럼: `id`, `engineer_no`(식별번호), `name`, `rank`(직위: 상무·이사·부장 등), `position`(직책, 원본에 없어 선택입력), `company`, 연락처, `region`, `address`, `employment_status`(재직/퇴직/비활성 — 소프트 삭제 역할), `joined_date`/`retired_date`, `memo`, `is_favorite`.
- 분야: `engineer_specialties`(마스터 12종: 건축·토목·기계·전기·통신·소방·안전·품질·공정·사업관리·조경·기타) + `engineer_contact_specialties`(다중 지정 junction). **첨부 엑셀 실측 분야는 건축/토목/안전/기계 4종만 등장**하지만 마스터는 이미 12종을 수용한다.
- `overtime_employees`(팀 내부 11명)와는 **의도적으로 분리된 별도 테이블** — 용도·규모·수명주기가 다르다는 이유가 문서화되어 있음. 이번 기능의 "기술인"은 외부 인력이므로 `engineer_contacts`가 자연스러운 마스터 후보.
- **참고**: `engineer_contacts`가 프로젝트를 참조하는 FK는 현재 없다. 즉 "이 기술인이 어느 프로젝트에 참여했는지"를 조회하는 관계도 아직 존재하지 않는다.

## 3. 연장근무의 "전월 21일~당월 20일" 기간 계산 (Confirmed — 그대로 재사용 가능)

파일: `lib/overtime/summary.ts`

```typescript
interface PayPeriodDay { year: number; month: number /* 0-indexed 실제 달력월 */; day: number; dateStr: string }

payPeriodDays(year, month): PayPeriodDay[]   // month는 "기간 라벨 월"(예: payPeriodDays(2026,8) → 2026-07-21~2026-08-20)
payPeriodRange(year, month): { start, end }  // DB gte/lte 조회용 문자열
currentPayPeriod(): { year, month }          // 오늘 기준 "이번 기간" 라벨. 21일 이후면 다음 달 라벨로 자동 이월
```
- 이미 순수 함수(year/month 입력 → 배열/문자열 출력)이고 연장근무 도메인에 결합돼 있지 않다 — **그대로 import해 재사용 가능**, 복제 불필요.
- 이 기간 개념은 "8단계 완료 후 사용자 요청으로 변경"됐다고 `docs/overtime.md`에 명시돼 있음 — 즉 이 회사의 실제 급여/마감 관행이며 이번 기능이 요구하는 "전월 21일~당월 20일"과 **동일한 규칙**.
- 연간 통합 기간(전년도 12/21~해당연도 12/20)을 반환하는 함수는 **아직 없음** — 신규 작성 필요(단, `payPeriodDays`를 12회 이어붙이면 파생 가능).
- 월 라벨 이동(‹›) 로직은 "라벨 연/월을 1씩 증감"이라 단순 — 그대로 재사용 가능.

## 4. 기존 출력(Export) 기능 분석 (Confirmed)

### 4.1 실제 xlsx 바이너리 생성 라이브러리 — **코드베이스에 없음**
`package.json` 의존성에 `exceljs`/`xlsx`/`sheetjs` 계열이 전혀 없다. 있는 것은 `pdfjs-dist`(PDF **읽기** 전용, 생성 아님)뿐.

### 4.2 실제로 쓰는 "출력" 두 가지 패턴
| 패턴 | 사용처 | 구현 |
|---|---|---|
| CSV 다운로드 | 프로젝트 List, 연장근무 출력(`/overtime/print`) | BOM + Blob + `<a download>` — 서버 없이 클라이언트에서 문자열 조립. 엑셀에서 바로 열림 |
| 브라우저 인쇄 | 연장근무 출력 | `window.print()` + `globals.css`의 `.no-print`(`@media print{display:none}`)로 버튼/내비 숨김. Ctrl+P → "PDF로 저장"이 PDF 출력 대체 |

`docs/overtime.md` 8단계 결정사항에 "PDF/Excel 파일 생성은 만들지 않았다 — 마스터 프롬프트가 향후 과제로 명시" + "API route(`app/api/overtime/export/route.ts`)는 만들지 않았다"고 명시. 즉 **진짜 xlsx 바이너리 생성은 이 코드베이스 전체에서 한 번도 구현된 적이 없다.**

### 4.3 HWPX 생성 — Excel과 무관한 별개 파이프라인
`app/api/hwpx/route.ts`: 주간/월간 업무보고 전용. `kordoc`+`adm-zip`+`@xmldom/xmldom`로 `lib/templates/*.hwpx` 템플릿의 `Contents/section0.xml`을 파싱해 표 셀(`hp:tc`)을 텍스트로 채워 재압축한다. **한글(HWP) 포맷 전용 로직**이라 Excel 셀 음영·병합 조작에는 재사용 불가 — 다만 "템플릿 파일을 열어 좌표별로 셀을 채우고 다시 저장"하는 접근 자체는 Excel 템플릿에도 그대로 적용 가능한 패턴.

### 4.4 결론
Excel 출력을 실제로 구현하려면 **새 의존성 도입이 필요하다** (→ `05-export-spec.md`에서 후보 검토).

## 6. 권한 구조 분석 (Confirmed)

- `allowed_users.menu_permissions`(jsonb): 메뉴 키별 `'none'|'read'|'write'` 3단계. 키 목록은 `lib/menuConfig.ts`의 `RESTRICTABLE_MENU_ITEMS`(현재 9개 메뉴 — `projects`, `weekly`, `overtime`, `leave`, `engineers`, `sites`, `trip`, `web`, `proposal_db`).
- `none`=사이드바 숨김, `read`=조회만(수정 UI 비활성화), `write`=전체. **UI 레벨 제어**이며 RLS(DB 단 권한)는 아니다.
- `ADMIN_EMAILS`(환경변수) 목록의 관리자는 `allowed_users`에 없어도 항상 전체 쓰기 권한.
- **"월 마감"처럼 쓰기 권한과 별개인 개별 액션 전용 권한 레벨은 현재 존재하지 않는다.** write 권한자는 곧 그 메뉴의 모든 CRUD가 가능하다는 뜻이라, "일반 쓰기 사용자는 되고 마감/마감취소는 관리자만" 같은 세분화된 권한 개념이 없다 (→ Open Question, `02-requirements.md`).
- RLS는 모든 테이블에 "인증된 사용자 전체 접근"만 걸려 있다 — 접근 제어는 layout(Server Component)에서 처리.

## 7. 첨부 엑셀 분석 (`commute_sample.xlsx`) (Confirmed — openpyxl로 직접 파싱)

> 분석 방법: `openpyxl`로 시트/병합/서식/색상/인쇄설정을 직접 읽음(화면 캡처 추정 아님). 파일은 단일 시트.

### 7.1 시트 개요
- 시트명 `2025년` 1개, 제목(A1) = `"서울사무실 근무현황(2025.11월)"`.
- 범위 A1:NQ238 (381열 × 238행). 틀고정: `E4` — **A~D열(4개) + 1~3행(3개) 고정**. 확대/축소 85%.
- 폰트: 맑은 고딕. 제목 16pt, 헤더(2행) 10pt bold, 데이터 10pt. 전 셀 얇은(thin) 테두리, 세로 가운데 정렬.

### 7.2 열 구성 (항목 1, 3 — Confirmed)
| 열 | 헤더 텍스트 | 폭 | 설명 |
|---|---|---|---|
| A | `프로젝트` | 20.75 | 프로젝트명 |
| B | `구분` | 4.75 | 실측값은 **`단장`뿐** (팀장/부단장 등 다른 값 없음) |
| C | `직종` | 4.875 | 실측값 `건축`(136)/`토목`(35)/`안전`(15)/`기계`(1) — 데이터 오기로 `단장`이 1건 들어간 사례 있음(입력 실수) |
| D | `성명` | 8.375 | |
| E~NP | 날짜(1~13개월치) | 각 3.125 | 아래 7.3 참고 |
| NQ | `비고` (줄바꿈 포함 2줄 헤더) | 11.875 | 가운데 정렬, 내용 있는 셀만 `wrapText=True` |

**중요(항목 4 — Confirmed): 날짜 범위가 "전월 21일~당월 20일" 달력이 아니라 전체 캘린더 연속이다.** 헤더 2행 병합 블록은 `2024년 12월`(21~31일, 11일)부터 시작해 `2025년 1월`(1~31일) … `2025년 12월`(1~31일)까지 **총 376일**(2024-12-21~2025-12-31)을 한 줄로 이어 붙인 것 — 월 블록은 "그 달의 날짜 전체"이지 "21일~20일"로 잘려 있지 않다. 마지막 12월 블록도 20일에서 끊지 않고 31일까지 그려져 있다 (→ 연간 출력 범위에 대한 Open Question, `02-requirements.md` 참고).

### 7.3 프로젝트별 참여인원 행 구성 (항목 1, 2 — Confirmed)
- **병합셀이 아니다.** 프로젝트명(A열)·구분(B열)은 그 프로젝트의 **첫 번째 기술인 행에만 값이 있고, 같은 프로젝트의 나머지 행은 그냥 빈 셀**이다(수직 병합 아님 — 전체 시트의 병합 범위는 19개뿐이며 전부 1~3행 헤더용 + 원인불명 이상 병합 1건, 본문 데이터에는 병합이 전혀 없음).
- 실측 84개 프로젝트, 총 188개 기술인 행. 블록 크기 분포: 1명 23건 / 2명 24건 / 3명 31건 / 4명 6건 (평균 2.24명). **상한 4명은 표본 특성일 뿐 업무 규칙상 제한이 아니다.**
- 프로젝트 블록의 첫 행은 거의 항상 `구분='단장'`이지만 예외 1건 확인(`구분='단장 '`— 트레일링 공백, 데이터 정합성 문제).
- **단장 교체가 실제로 이렇게 표현된 사례 확인**: 같은 프로젝트 블록 안에 `구분='단장'` 행이 **2개** 존재하는 경우가 2건 있음(예: "프로젝트 A" — 기존 단장 → 변경 단장). 즉 기존 관행은 **덮어쓰지 않고 새 행을 추가**해 과거 단장 이름을 지우지 않는다 — 이번 기능의 "과거 기록 보존" 요구사항과 정확히 일치하는 실제 관행.
- 데이터 실제 영역은 4~191행(188개 유효 행 근처)이고, 192~238행은 **완전히 빈 버퍼 행**(같은 서식만 미리 깔아둔 여백) — 향후 프로젝트 추가를 대비한 사전 서식.

### 7.4 출근일 음영 방식 (항목 6 — Confirmed)
- 출근 표시는 **오직 셀 배경색(단색 채우기)뿐**이며, 대부분 **셀 값은 비어 있다**(값 없음 + 배경색만). 날짜 그리드 전체 88,360셀 중 4,996셀(5.6%)이 채워짐(theme 색 0, tint -15% — Excel 기본 팔레트의 옅은 회색, 브랜드 지정색 아님).
- **예외 — 셀에 텍스트가 함께 들어간 경우가 187건 확인됨**: `심사`(기술인 심사/면접 대기 145+30건), `교체`(5건), `취소`(3+3건), `재공고`(3건), `추석 연휴`(1건, 노란색 `FFFF00` 단색 강조 — 이 한 건만 회색이 아닌 별도 색상). 즉 **실무에서는 "비고" 개념을 날짜 그리드 안에 직접 짧은 단어로도 적어 넣고 있었다** — 프로젝트 단위 변경사항이 우측 비고칸이 아니라 해당 날짜 셀에 바로 기록된 사례가 훨씬 많다(187건 vs 비고칸 5건).
- **주말/공휴일 전용 서식은 사실상 없다.** 헤더 3행(요일 대신 일자 숫자만 있음 — 요일 텍스트 자체가 없음)에서 셀 배경이 칠해진 곳은 376일 중 3일뿐(2025-01-04~06, 그나마 1/6은 월요일이라 주말과도 안 맞음) — 실제 주말 108일 중 106일은 아무 표시가 없다. **이 3칸은 초기 작업 흔적으로 보이는 예외이지 체계적인 주말/공휴일 표시 규칙이 아니다** (→ 새 시스템은 주말/공휴일을 프로그램적으로 계산해 일관되게 표시해야 하며, 기존 엑셀의 표시 방식을 "재현할 확정 규칙"으로 볼 수 없음. Open Question 아님 — 기존에 규칙이 없었다는 것 자체가 Confirmed 사실).
- 하나의 이상 병합(`KE129:KN155`, 10개 날짜열×27개 인원행)이 발견됨 — 위치가 정확히 2025-10-03("추석 연휴" 노란 셀)과 겹친다. 연휴 기간 전체를 큰 블록 하나로 병합해 "추석 연휴"라는 라벨 하나로 퉁친 것으로 추정되는 **일회성 수작업 흔적**이며, 재현할 패턴이 아니라 예외로 분류.

### 7.5 비고란 (항목 8 — Confirmed)
- 실제 비고(NQ열) 사용 건수는 235개 데이터 행 중 **5건뿐**. 내용은 사용자가 나열한 카테고리(단장교체·공고취소 등)와는 결이 다르고, 실측은 "다른 프로젝트 수주로 인한 중도 이탈"(예: `프로젝트 B 수주(중도취소)`, `프로젝트 C 수주(중도취소)`) — 기술인이 다른 프로젝트에 배정되어 이 프로젝트에서 빠진 사례. 같은 문구가 관련된 기술인 행마다 반복 입력돼 있어(1건의 사건 → 여러 행에 동일 텍스트 수기 복사), **병합이나 참조 없이 단순 텍스트 반복**이 실무 관행이었다.
- wrapText는 내용이 있는 셀에만 켜져 있고 빈 셀은 기본값(off) — 즉 자동 줄바꿈이 아니라 콘텐츠가 들어갈 때 그때그때 적용된 것으로 보임.

### 7.6 인쇄 설정 (항목 9 — Confirmed, 그리고 중요한 공백)
- 용지: A4 세로(`orientation=portrait`, `paperSize=9`). 여백은 기본값(0.7/0.7/0.75/0.75 in).
- **인쇄영역(`print_area`) 지정 없음. 반복 인쇄 제목행/열(`print_title_rows/cols`) 지정 없음. 페이지 나누기(수동) 없음. 배율/페이지맞춤(`fitToWidth/Height`, `scale`) 전부 미지정. 머리글/바닥글 텍스트 없음.**
- 즉 **이 파일 자체에는 "출력에 필요한 인쇄 설정"이 하나도 저장돼 있지 않다.** 실무에서는 인쇄할 때마다 그때그때 범위를 선택해 인쇄했을 가능성이 높음 — 마스터 프롬프트가 요구하는 "가로 방향/반복 제목행/여백/배율" 등은 **기존 파일을 재현할 대상이 아니라 이번에 새로 설계해야 하는 요구사항**이라는 뜻.

### 7.7 병합셀·행높이·열너비 (항목 10 — Confirmed)
- 헤더 2~3행: 열 병합 12개(월별) + A/B/C/D열 각각 2~3행 세로 병합(라벨용) + NQ열 2~3행 병합. 본문(4행 이후)에는 병합 없음(위 7.3 참고).
- 데이터 행 높이 31.5(고정, 4~238행 전부 동일) — 프로젝트명 등 긴 텍스트의 줄바꿈을 감안한 높이로 추정. 헤더 1행 30, 2행 16.5, 3행은 기본값(미지정).
- 날짜 열 폭은 전 구간 동일 3.125(일자 숫자 두 자리가 겨우 들어가는 폭) — "가로 가상 스크롤" 요구사항이 왜 필요한지 실측으로도 뒷받침됨(376개 열이 전부 균일polje 좁은 폭으로 나열).

## 9. Open Question (이 문서에서 발견된, 코드/파일로 확인 불가능한 사항)

1. Project List에 재공고/변경공고/취소 여부 전용 필드가 없다 — 신규 컬럼을 `projects`에 추가할지, 출근부 쪽 신규 테이블에만 둘지 결정 필요.
2. "월 마감" 전용 권한 레벨이 현재 권한 체계에 없다 — 어떻게 표현할지(신규 권한 키? ADMIN_EMAILS 재사용? menu_permissions 값 확장?) 결정 필요.
3. 첨부 엑셀의 연간 날짜 범위가 실제로는 "당해 12/20 마감"이 아니라 "당해 12/31까지"로 그려져 있다 — 마스터 프롬프트의 "연간 마감 12/20" 규칙과 실물 엑셀 습관 중 무엇을 기준으로 삼을지 확인 필요.
4. 엑셀 실무에서는 프로젝트 변경사항(교체/취소/재공고 등)을 비고칸보다 **날짜 셀에 직접** 적는 경우가 압도적으로 많았다(187건 vs 5건) — 이 관행을 그대로 "특정 날짜에 종속된 이벤트"로 모델링할지, 전부 프로젝트 비고/변경이력으로 흡수할지 확인 필요.

이 4가지는 `02-requirements.md`에서 다시 다룬다.
