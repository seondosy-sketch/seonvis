# 연장근무 관리 (제안서팀)

> 제안서팀의 연장근무 입력·업무이력 관리·월말 행정업무 자동화를 위한 기능.
> 8단계로 나누어 단계별 코드 리뷰를 거치며 진행한다 — **임의로 다음 단계까지 구현하지 않는다.**

---

## 절대 변경하지 않는 원칙

UI와 데이터 구조는 분리한다. 사용자는 엑셀처럼 입력하지만, 실제 DB는 업무 1건 단위로 저장한다.

```
직원 1명 + 날짜 1개 + 프로젝트 1개 + 업무 1개 = Record 1개
```

예) 홍길동이 7월 3일에 화성여자교도소(18~20시)/서산의료원(20~21시)/충주문화원(21~24시) 3건을 했다면
DB에는 Record 3건이 저장되고, 화면(월간 그리드 셀)에는 `6h (3)`만 보인다.
총 시간·건수는 컬럼으로 저장하지 않고 항상 그 날의 Record들로부터 계산한다.

---

## "한 달"의 기준 (3~8단계 전체 완료 후 사용자 요청으로 변경)

이 기능에서 "한 달"은 달력 월(1일~말일)이 아니라 **전달 21일 ~ 이번달 20일**(급여 마감 기준)이다.
예: "7월"로 표시되는 기간은 실제로는 `2026-06-21` ~ `2026-07-20`. 이유는 급여 계산 마감일에 맞춘
회사 관행이며, 그리드·대시보드·출력 세 화면 모두 이 기준을 똑같이 쓴다.

- **`lib/overtime/summary.ts`에 공용 함수로 구현**했다: `payPeriodDays(year, month)`가 그 기간에 속한
  모든 날짜를 `{year, month, day, dateStr}` 배열로 반환하고, `payPeriodRange`는 DB 조회용 시작/종료
  문자열만, `currentPayPeriod()`는 오늘 날짜 기준으로 "이번 기간"의 라벨(year, month)을 구한다
  (21일 이후면 다음 달 라벨로 넘어간다 — 예: 7/25는 "8월" 기간(7/21~8/20) 소속).
- **기간이 두 달에 걸치므로** 그리드(`MonthGrid`)·대시보드 일별 차트·출력 페이지 모두 "연/월 하나 +
  일수"로 날짜를 만들지 않고, `PayPeriodDay[]` 배열(각 날짜의 실제 연/월/일 포함)을 그대로 순회한다.
  `MonthGrid`는 이제 `year`/`month`를 받지 않고 `days: PayPeriodDay[]`만 받는다 — 어떤 날짜 범위를
  주든 그리는 것만 신경 쓰면 되게 단순해졌다.
- **달력 월이 바뀌는 첫 칸에는 작게 "M월"을 표시**하고 왼쪽 테두리를 굵게 처리했다(그리드, 대시보드
  일별 차트, 출력 표 전부 동일). 안 그러면 "1일"이 6월 1일인지 7월 1일인지 구분이 안 된다. 화면
  상단에는 항상 `(6/21 ~ 7/20)`처럼 실제 날짜 범위를 라벨 옆에 괄호로 병기해 "7월"이라는 표시가
  달력 7월 전체를 뜻하는 게 아님을 분명히 했다.
- **월 이동(‹ ›) 로직은 그대로**다 — 라벨(year, month)을 1씩 증감하기만 하면 되므로, 기존 "달력 월
  이동" 코드를 바꿀 필요가 없었다.

## 그리드 열 너비 / 일괄 입력 (사용자 요청으로 추가)

- **그리드 열 너비를 컨테이너 폭에 맞춰 자동으로 나누도록 변경**했다(`MonthGrid.tsx`). 이전에는 날짜
  컬럼마다 `minWidth: 44`를 고정해서 날짜가 30일이면 표 전체 너비가 화면보다 넓어져 가로 스크롤이
  생겼다. `<table style={{ tableLayout: 'fixed', width: '100%' }}>` + `<colgroup>`으로 직원 컬럼만
  고정 폭(100px)을 주고 나머지 날짜 컬럼은 폭을 지정하지 않아, 브라우저가 남은 공간을 날짜 수만큼
  균등하게 나눠 갖게 했다 — 그래서 기간이 며칠이든(28~31일) 항상 가로 스크롤 없이 한 화면에 다 보인다.
- **일괄 업무 입력 기능 추가**(`_components/BulkWorkRecordModal.tsx`). "같은 프로젝트·같은 업무를
  여러 날에 반복 입력"하는 상황(예: 이번 주 내내 같은 현장 점검)을 위해, 직원·프로젝트·업무내용·
  시작~종료시간·비고는 한 번만 입력하고 **날짜만 여러 개(체크 형태 pill) 선택**해서 한 번에 저장한다.
  "전체 선택"/"평일만"/"전체 해제" 버튼으로 빠르게 고를 수 있다.
  - 저장은 선택한 날짜 수만큼 행을 만들어 **`insert(rows)` 한 번**으로 처리한다(날짜마다 별도
    요청을 보내지 않음).
  - 1단계 핵심 원칙("직원 1명 + 날짜 1개 + 프로젝트 1개 + 업무 1개 = Record 1개")은 그대로다 — 이
    모달은 그 레코드를 여러 개 한 번에 만드는 입력 편의 기능일 뿐, 저장 단위 자체를 바꾸지 않는다.
  - 헤더의 "일괄 입력" 버튼에서 열고, 프로젝트 선택지는 `WorkRecordForm`과 동일하게 진행중 프로젝트만
    보여준다.

## 직원별 기본업무내용 (8단계 완료 후 사용자 요청으로 추가 — 1차)

향후 근무입력(WorkRecordForm/BulkWorkRecordModal)에서 업무내용을 자유 텍스트 대신 직원별
드롭박스로 고를 수 있게 하려는 작업의 **1차 단계**다. 이번 단계에서는 "그 드롭박스의 기초자료"인
직원별 업무내용 목록을 직원 관리 화면에서 등록/삭제하는 기능만 추가했다 — **근무입력 화면의
드롭박스 연동은 아직 하지 않았다** (사용자가 "먼저"라고 범위를 명시적으로 한정함). `task_description`은
지금도 여전히 자유 텍스트 입력이다.

- **새 테이블 `overtime_employee_tasks`** 추가(`supabase/migration_overtime_employee_tasks.sql`).
  `overtime_employees`의 하위 테이블로, `(employee_id, task_name)` UNIQUE — 같은 직원에게 같은
  업무명을 중복 등록할 수 없다. `overtime_work_records`와 달리 이력 데이터가 아니라 "제안 목록"일
  뿐이므로 FK는 `ON DELETE RESTRICT`가 아니라 `ON DELETE CASCADE`로 걸었다.
- **`EmployeeManagerModal.tsx`에 직원별 업무내용 관리 UI 추가**: 각 직원 행에 "기본업무내용 ▼"
  토글 버튼을 달아 펼치면 그 직원의 등록된 업무내용을 칩(chip) 형태로 보여주고, 칩의 ✕로 삭제,
  입력창 + "추가" 버튼으로 새 업무명을 추가한다. 목록은 토글로 처음 펼칠 때만 조회한다(직원마다
  목록을 미리 다 불러오지 않음 — 직원 수만큼 불필요한 쿼리를 늘리지 않기 위함).
- **`lib/overtime/types.ts`에 `EmployeeTask` 타입 추가**. 다른 타입들과 동일하게 DB 컬럼명과 1:1.
- **다음 단계(미착수)**: `WorkRecordForm`/`BulkWorkRecordModal`의 "업무내용" `<input>`을 선택된
  직원의 `overtime_employee_tasks`를 옵션으로 하는 드롭박스(+ 목록에 없는 값은 직접 입력 가능한
  콤보박스 형태 권장 — 기존에 자유 텍스트로 쌓인 데이터와의 호환을 위해)로 바꾸는 작업.

## 프로젝트 시작일/종료일 (8단계 완료 후 사용자 요청으로 추가)

- **`overtime_projects`에 `start_date`/`end_date` (date, nullable) 컬럼 추가**
  (`supabase/migration_overtime_project_dates.sql`). 기존 프로젝트가 이미 있으므로 nullable —
  기간을 아직 정하지 않은 프로젝트도 허용한다.
- **`ProjectManagerModal.tsx`에 시작일~종료일 `<input type="date">` 두 개 추가**. 다른 필드와
  동일하게 `onBlur` 즉시저장이고, 비우면 null로 저장한다. 날짜 입력 두 개가 들어가면서 모달
  너비를 520 → 680으로 넓혔다.
- **향후 계획 (사용자 예고, 미착수)**: 프로젝트 목록을 이 모달에서 직접 입력하는 대신 **입찰 현황
  "프로젝트 List"(`projects` 테이블)에서 가져오는 방안**이 예정되어 있다. 그때 `overtime_projects`가
  `projects`를 참조/동기화하는 구조로 바뀔 수 있으므로, 이 모달에 기능을 더 얹기 전에 그 작업과의
  순서를 먼저 정하는 게 좋다.

## 프로젝트별 담당직원 (8단계 완료 후 사용자 요청으로 추가)

향후 **프로젝트별 인원을 나열해 근무일을 표기**하는 화면의 기초자료다. 이번 단계에서는 배정
데이터의 등록/해제만 만들었다 — 인원별 근무일 나열 화면은 아직 없다.

- **새 테이블 `overtime_project_members`** (`supabase/migration_overtime_project_members.sql`).
  `(project_id, employee_id)` UNIQUE 연결 테이블. 실제 근무 이력이 아니라 "배정" 정보이므로
  FK는 `ON DELETE CASCADE` — `overtime_employee_tasks`와 같은 이유.
- **`ProjectManagerModal.tsx`에 "담당직원 ▼" 토글 추가**: 행을 펼치면 직원 체크박스 목록이
  나오고 체크/해제 즉시 저장된다. 버튼에 현재 배정 인원수를 표기한다.
  - 배정 데이터는 처음에 **전부 한 번에 불러온다** (직원 목록 + members 테이블 통째). 토글
    버튼마다 인원수를 바로 보여줘야 해서, 기본업무내용(`EmployeeManagerModal`)처럼 펼칠 때
    lazy 조회하는 방식과 달리 갔다 — 테이블이 프로젝트×직원 수준으로 작아 부담이 없다.
  - 체크 목록에는 재직 중인 직원만 보여주되, **이미 배정된 퇴사자는 "(퇴사)" 표기와 함께 남겨**
    해제할 수 있게 했다.
- **`lib/overtime/types.ts`에 `ProjectMember` 타입 추가**.
- 날짜 입력(시작일/종료일)에 담당직원 버튼까지 행이 길어져 **모달 너비를 680 → 780으로** 넓혔다.

## 프로젝트별 그리드 보기 (8단계 완료 후 사용자 요청으로 추가)

그리드 페이지(`/overtime`)에 **"프로젝트별" 보기를 추가하고 기본 보기로** 했다. 기존 직원별
보기(MonthGrid)는 없애지 않고 우측 상단 토글(프로젝트별/직원별)로 남겨뒀다 — 직원별 총합을
한눈에 보는 용도는 여전히 유효해서.

```
| 프로젝트명       | 담당직원 | 6/21 | 6/22 | … | 7/20 | 합계 |
| 화성여자교도소    | 김○○   |  2h  |      | … |      | 12h |
| (6/21~7/15)    | 이○○   |      |  2h  | … |  3h  | 16h |
| ○○체육센터      | 김○○   |      |  3h  | … |  2h  |  8h |
```

- **`_components/ProjectGrid.tsx`** — 행이 "프로젝트 × 담당직원"(프로젝트명 셀은 rowSpan으로
  묶음), 열이 날짜, 마지막 열이 행 합계. 셀 값은 같은 `overtime_work_records`를
  "프로젝트 × 직원 × 날짜"로 합산한 것 — 저장 단위(1단계 원칙)는 그대로다.
- **프로젝트 기간(start_date/end_date) 반영**: 기간 밖 날짜 셀은 회색 + 클릭 불가로 막아
  "프로젝트 기간 내에만 근무시간을 입력"하게 했다. 단, 기간 밖인데 이미 기록이 있는 셀
  (기간을 나중에 좁힌 경우)은 값을 보여주고 클릭도 허용해 수정할 수 있다. 날짜 비교는
  `YYYY-MM-DD` 문자열 사전순 비교로 처리(UTC 파싱 버그 회피).
- **행 구성**: 진행중 프로젝트 + (종료됐지만 이 기간에 기록이 있는 프로젝트)만 표시.
  각 프로젝트의 행 직원은 담당 배정(`overtime_project_members`)된 직원 + 배정은 안 됐지만
  그 프로젝트 기록이 있는 직원 — 후자를 빼면 이미 입력된 시간이 화면에서 사라져 보인다.
  담당직원이 하나도 없는 프로젝트는 "프로젝트 관리에서 담당직원을 체크하세요" 안내 행 하나.
- **셀 클릭 → 기존 WorkRecordModal 재사용**: 새 모달을 만들지 않고, `defaultProjectId`를
  넘겨 업무 추가 폼에 그 프로젝트가 미리 선택되게만 했다(`WorkRecordModal`/`WorkRecordForm`에
  optional prop 추가). 모달은 여전히 그 직원·그 날짜의 전체 기록(다른 프로젝트 포함)을 보여준다 —
  같은 날 다른 프로젝트 기록과의 맥락을 잃지 않기 위해.
- `page.tsx`는 이제 summaries(직원별 보기용)와 별도로 **원본 `records` 배열을 state로 유지**해
  ProjectGrid에 넘긴다. 프로젝트 목록과 담당 배정은 `loadProjects`에서 함께 불러온다.

## 셀 팝오버 입력창 (8단계 완료 후 사용자 요청으로 추가)

프로젝트별 그리드의 셀 클릭 시, 큰 모달 대신 **셀 근처에 뜨는 작은 팝오버 입력창**
(`_components/OvertimeEntryPopover.tsx`)이 열린다. 프로젝트명/직원명/근무일은 셀에서 이미
정해져 있으므로 헤더에 자동 표시만 하고, 사용자는 업무내용과 근무시간 유형만 고른다.

- **업무내용은 드롭다운** — 선택지는 그 직원의 기본업무내용(`overtime_employee_tasks`).
  "직접 입력..." 옵션으로 자유 텍스트도 가능하다. 기존 기록의 업무내용이 목록에 없으면
  직접 입력으로 매핑해 값이 사라지지 않게 했다. (기본업무내용 기능이 여기서 처음 실사용됨)
- **근무시간 유형 3가지**:
  - `2시간`: 18:00~21:00, 휴게 1시간 → 인정 2시간
  - `3시간`: 18:00~22:00, 휴게 1시간 → 인정 3시간
  - `기타`: 시작/종료/휴게시간 직접 입력 → **인정 = 종료 − 시작 − 휴게, 1시간 단위 절삭(내림)**
    (예: 18:00~23:30, 휴게 1시간 → 계산 4.5시간 → 인정 4시간). 계산은
    `lib/overtime/time.ts`의 `calculateRecognizedHours` — 기존 `calculateHours`(식사시간
    자동 차감)와 별개 함수다. 절삭 전/후를 함께 반환해 폼이 "계산 4.5시간 → 인정 4시간"을
    실시간으로 보여준다. 인정시간 1시간 미만이면 저장을 막는다.
- **`overtime_work_records.break_hours` 컬럼 추가** (`supabase/migration_overtime_break_hours.sql`,
  numeric(3,1) nullable). 팝오버 저장분은 휴게시간 명시값이 들어가고, **null은 기존 방식
  레코드**(저장 시점 식사시간 자동 차감)를 뜻한다 — 기존 데이터는 건드리지 않는다.
  기존 `WorkRecordForm`/`BulkWorkRecordModal`은 여전히 자동 차감 방식으로 저장한다(null).
- **같은 셀에 기록이 여러 건이면** 팝오버 상단에 목록을 보여주고 골라서 수정하거나
  "+ 새 업무 추가"로 새 건을 만든다 — "업무 1건 = Record 1건" 원칙 유지.
- 기존 기록을 열면 시간 값으로 유형을 역추론한다(18:00~21:00·2h → "2시간" 등, `detectTimeType`).
- **직원별 보기(MonthGrid)의 셀 클릭은 기존 WorkRecordModal 그대로** — 그 보기의 셀은
  프로젝트가 특정되지 않아(하루 총합) 팝오버의 "프로젝트 자동 표시" 전제가 안 맞는다.

### 3단계 결정사항 — 레이아웃

원본 요구사항의 ASCII 예시(`홍길동 2 3 6 4 ...`)를 보면 "좌측 직원 목록"과 "우측 월간 캘린더"는
독립된 두 패널이 아니라 **표 하나**다 — 직원명이 표의 첫 컬럼(행 헤더), 날짜가 나머지 컬럼(열 헤더)인
스프레드시트 구조. 그래서 `EmployeeList`를 따로 만들지 않고, `MonthGrid`의 첫 컬럼을
`position: sticky; left: 0`으로 고정해 가로 스크롤에도 직원명이 항상 보이게 했다.
행 헤더(직원명)·열 헤더(날짜)가 각각 sticky라 세로/가로 스크롤 모두 헤더가 고정된다.

이번 단계는 **틀만** 만들었다 — `overtime_employees`에서 재직 중인 직원만 정렬순서대로 불러와
행으로 그리고, 선택한 월의 날짜만큼 열을 그린다. 셀은 아직 비어 있다(4단계에서 `overtime_work_records`
집계로 채움). 클릭도 아직 없다(5단계). 헤더의 "직원 관리"/"프로젝트 관리" 버튼은 자리만 잡아두고
`disabled` 상태로 뒀다(6단계에서 연결).

`overtime_employees` 테이블이 아직 Supabase에 없다면(2단계 마이그레이션 미적용) 빨간 경고 박스로
안내하고 빈 그리드를 보여준다.

### 4단계 결정사항 — 월간 그리드 데이터

- 선택한 월의 `[1일, 말일]` 범위로 `overtime_work_records`를 한 번에 조회하고, `lib/overtime/summary.ts`의
  `summarizeByEmployeeAndDate()`로 `employee_id + work_date` 키의 `DailySummary` 맵을 만든다.
  합계·건수는 이 함수에서만 계산한다 — 컴포넌트가 직접 더하지 않는다.
- `overtime_work_records`는 이번 단계에서도 `select('*')`로 전체 컬럼을 가져온다. 그리드는 `total_hours`/`record_count`만
  쓰지만, `DailySummary.records`가 온전한 `WorkRecord[]`여야 5단계 모달이 그대로 재사용할 수 있기 때문 — 5단계에서
  쿼리를 다시 만들 필요가 없다.
- 셀 표시 규칙: `record_count > 1`일 때만 우측 상단에 `(건수)` 배지를 단다. 1건이면 시간만 표시(예: `6h`), 기록이 없는
  날은 빈 셀. `formatHours()`는 정수 시간이면 `6h`, 소수면 `6.5h`로 표시.
- 월 이동 시 직원 목록은 다시 불러오지 않고 `overtime_work_records`만 재조회한다(직원 수가 적고 월과 무관하기 때문).
  그리드 전체를 숨기지 않고 상단에 "집계 불러오는 중..."만 표시해 화면이 깜빡이지 않게 했다.

### 5단계 결정사항 — 셀 클릭 모달

- **기록이 없는 셀도 클릭 가능**하다. 마스터 프롬프트가 "셀을 클릭하면 모달이 열린다"고만 했고,
  빈 날짜에 업무를 추가하는 흐름(6단계)도 결국 셀 클릭에서 시작하므로 빈 셀도 처음부터 클릭 가능하게 했다.
  `summary`가 없으면 총 연장 0시간 + "등록된 업무가 없습니다"로 표시한다.
- **프로젝트 이름 조회를 5단계에서 추가**했다. 모달이 각 업무 줄에 "프로젝트 {이름}"을 보여줘야 하는데
  `WorkRecord`는 `project_id`만 갖고 있기 때문. `overtime_projects`를 전체(종료 포함) 조회해서
  `id → name` 맵을 만든다 — 프로젝트 관리(등록/수정/종료 CRUD)는 여전히 6단계 몫이고, 여기서는 이름을
  읽기만 한다.
- **[업무 추가] 버튼은 자리만 잡아두고 `disabled`** — 3단계의 "직원 관리"/"프로젝트 관리" 버튼과 같은 패턴.
  실제 추가 폼과 저장 로직은 6단계.
- 날짜 헤더(`7월 3일`)는 `new Date("YYYY-MM-DD")`로 파싱하지 않고 문자열을 직접 분해한다 —
  `docs/conventions.md`가 경고하는 UTC 파싱 버그(한국 시간대에서 하루 전으로 밀리는 문제)를 피하기 위함.

### 6단계 결정사항 — 업무 CRUD / 프로젝트·직원 관리

- **`queries.ts`는 만들지 않았다.** 1단계 로드맵에는 있었지만, 이 코드베이스의 기존 페이지들
  (`projects/page.tsx`, `dashboard.tsx`)이 전부 컴포넌트 안에서 `supabase.from(...)`을 직접 호출하는
  방식이라 별도 쿼리 레이어를 두는 게 이 프로젝트에는 없는 관례를 새로 만드는 것이었다. 그래서
  `WorkRecordForm`/`ProjectManagerModal`/`EmployeeManagerModal`도 각자 `createSupabaseBrowserClient()`를
  호출해 직접 쿼리한다.
- **시간 계산은 자정을 넘겨도 되게** `lib/overtime/time.ts`의 `calculateHours()`가 처리한다.
  `<input type="time">`(00:00~23:59 제한) 대신 일반 텍스트 입력을 쓰고, "24:00", "25:30"처럼 24시간을
  넘는 표기를 그대로 허용한다 — 종료시간이 시작시간보다 늦지 않으면 저장을 막고 에러를 보여준다.
- **식사시간 1시간 자동 차감** (6단계 완료 후 사용자 요청으로 추가): 시작시간이 **12시 또는 18시**면
  그 구간에 점심/저녁 식사시간이 낀 것으로 보고 계산된 시간에서 1시간을 뺀다 (예: 18:00~21:00 → 3시간이
  아니라 2시간). 시(hour) 값만 보고 판단하므로 18:05, 18:30도 트리거된다 — 분 단위까지 정확히 "18:00"일
  필요는 없다. **원래 계산된 시간이 식사시간(1시간)보다 길지 않으면(1시간 이하, 예: 18:00~19:00) 차감하지
  않고 원래 시간을 그대로 저장한다** — 정확히 1시간 일한 걸 식사시간으로 전부 상쇄해 0시간으로 만들지
  않기 위함 (처음에는 "1시간보다 짧으면"만 예외로 뒀다가, 18:00~19:00이 0시간이 되는 게 이상해서 경계를
  "1시간 이하"로 넓혔다).
  차감은 **레코드(업무 1건) 단위**로 적용된다 — 하루에 여러 프로젝트를 이어서 입력해도 각 레코드의
  시작시간을 기준으로 개별 판단하므로, 식사시간이 포함된 구간을 하나의 레코드로 입력하든 별도
  레코드로 나눠 입력하든 하루 총합은 같다.
  - `WorkRecordForm`의 시작시간 기본값을 `"18:00"`으로 바꿨다(연장근무는 대개 정규 근무 종료 직후인
    18시부터 시작하므로) — 물론 직접 수정 가능하다.
- **[업무 추가] 버튼을 누르면 목록 밑에 폼이 나타난다** (별도 모달을 겹치지 않음). 프로젝트 드롭다운은
  기본적으로 진행중 프로젝트만 보여주되, **수정 중인 기록이 이미 종료된 프로젝트를 가리키면 그 프로젝트도
  선택지에 넣는다** — 안 그러면 수정 폼을 열었을 때 드롭다운이 빈 값으로 보인다.
- 저장/삭제가 끝나면 모달은 스스로 목록을 다시 그리지 않고 **부모(page.tsx)의 `onSaved`/`onChange`를 호출해
  `overtime_work_records`/`overtime_employees`/`overtime_projects`를 다시 조회**하게 한다. 그리드 셀 값,
  모달 안의 목록, 관리자 모달 목록이 모두 같은 소스(재조회)에서 나오므로 서로 어긋날 일이 없다.
- **프로젝트/직원 삭제는 참조하는 `overtime_work_records`가 없을 때만 허용**한다(FK `ON DELETE RESTRICT`).
  Postgres가 `23503`(foreign_key_violation)을 돌려주면 "먼저 종료/퇴사 처리하라"는 안내 문구로 바꿔서 보여준다 —
  이미 쓰인 프로젝트·직원은 지우는 게 아니라 상태만 바꾸는 것이 2단계에서 정한 원칙이기 때문.
- **직원/프로젝트 목록은 셀 편집 즉시 저장**(입력 필드 `onBlur`, 체크박스 `onChange`)한다 — 별도 "저장" 버튼을
  누르지 않아도 되게 해서 "엑셀처럼 최소 클릭"이라는 개발 원칙을 지켰다.

### 7단계 결정사항 — 대시보드

- **`/overtime/dashboard`를 별도 라우트로 분리**했다(그리드 페이지와 다른 `page.tsx`). 데이터 성격이 다르고
  (그리드는 "선택한 달", 대시보드는 항상 "이번 달"), 6개 섹션이 한 화면에 다 있으면 그리드 페이지가
  너무 무거워지기 때문. 두 페이지 헤더에 서로 오가는 링크(`대시보드` / `← 그리드로`)만 달았다.
- **월 이동 없음** — 마스터 프롬프트가 "이번 달"이라고 명시했으므로 그리드처럼 월을 넘기는 UI를
  만들지 않았다. 필요해지면(예: 지난달 비교) 그때 추가한다.
- **집계 함수를 `lib/overtime/summary.ts`에 추가**했다(`monthRange`, `sumHoursByEmployee`,
  `sumHoursByProject`, `sumHoursByDate`). 이유: 8단계(월말 출력)의 "직원별 연장근무 내역"과
  "프로젝트별 투입내역"이 정확히 같은 집계이므로, 대시보드에서만 쓰는 컴포넌트 안에 묻어두면
  8단계에서 또 계산하게 된다. `page.tsx`의 로컬 `monthRange`도 이 공용 함수로 교체했다.
- **`WorkRecord`에 `created_at`을 추가**했다(`lib/overtime/types.ts`). DB 컬럼은 2단계부터 있었지만
  타입에는 없었다 — "최근 입력 내역"이 언제 등록됐는지로 정렬해야 처음으로 이 컬럼이 필요해졌다.
- **차트는 `dataviz` 스킬 규칙을 따르되, 이 앱의 기존 색만 재사용**했다: 새 브랜드 팔레트를 들여오지
  않고 이미 이 코드베이스가 쓰던 6색(amber/blue/green/orange/violet/red, `docs/conventions.md`에 있는
  색들)을 고정 순서로 쓰고, `scripts/validate_palette.js`로 CVD 안전성을 확인했다(PASS, 두 색은 배경
  대비가 낮아 "직접 라벨 필요" 경고 — 그래서 범례에 이름+퍼센트를 항상 같이 적었다).
  - **직원별/프로젝트별 연장시간**: 값(크기) 하나만 비교하는 단일 계열이라 amber 한 색으로 막대 길이만 다르게.
  - **일별 연장시간**: 같은 이유로 세로 막대 하나, 축 라벨(날짜)만 표시하고 막대마다 숫자를 적지는 않음
    (막대마다 값을 적지 말라는 규칙) — 정확한 값은 `title` 속성(마우스 오버)으로.
  - **프로젝트별 투입 비율**: 프로젝트 "정체성"을 구분해야 하므로만 색을 쓴다. 색은 고정 순서, 7개
    이상이면 "기타"로 묶어 색이 무한히 늘어나지 않게 했다. 범례에 색상 점 + 이름 + 퍼센트를 항상 표시해
    색만으로 구분하지 않게 했다.

### 8단계 결정사항 — 출력

- **PDF/Excel 파일 생성은 만들지 않았다.** 마스터 프롬프트가 "향후 PDF 및 Excel 출력도 지원한다"라고
  명시적으로 미래 과제로 남겨뒀고, 지금 만들면 새 의존성(엑셀/PDF 생성 라이브러리)이 필요하다. 대신
  지금 있는 것만으로 되는 두 가지로 "출력"을 채웠다:
  1. **CSV 다운로드** — `projects/page.tsx`의 `exportCsv`와 완전히 같은 방식(BOM + Blob + 다운로드 링크).
     엑셀에서 그대로 열리므로 "Excel 출력"의 실질적인 대체재가 된다.
  2. **브라우저 인쇄**(`window.print()`) — Ctrl+P → "PDF로 저장"을 고르면 그대로 PDF가 된다. 화면의
     버튼/내비게이션은 `globals.css`에 추가한 `.no-print`(`@media print { display: none }`)로 인쇄 시 숨긴다.
     `docs/conventions.md`가 명시한 유일한 예외(전역 CSS 클래스 허용)를 그대로 따랐다.
- **API route(`app/api/overtime/export/route.ts`)는 만들지 않았다.** CSV 생성은 순수 클라이언트 계산이라
  서버가 필요 없다. 1단계 로드맵에 있던 항목이지만, 실제로 필요해지는 시점은 "진짜" xlsx/PDF 바이너리를
  서버에서 만들 때(향후 과제)이므로 지금은 만들지 않는다.
- **월 선택 가능** — 대시보드(7단계)는 "이번 달"로 고정했지만, 출력은 "월말에 지난달 것을 뽑는" 용도가
  많으므로 그리드처럼 ‹ › 로 어느 달이든 조회할 수 있게 했다.
- **직원별/프로젝트별 내역은 화면에서는 그룹으로 묶어 보여주고, CSV는 평평한(flat) 표로 내려받는다.**
  화면은 "직원명 · 합계 Nh" 아래에 그 직원의 기록만 모아 보여주는 게 읽기 편하고, CSV는 이름이 매 행에
  반복되는 평평한 구조가 엑셀에서 피벗/필터하기에 더 낫다 — 같은 데이터를 용도에 맞게 다르게 표현한
  것뿐, 계산을 두 번 하지는 않는다(둘 다 같은 `records` 배열에서 파생).
- **전체 월간 집계표에는 대시보드에 없던 합계 행/열을 추가**했다. 그리드(`MonthGrid`)는 편집용이라 합계가
  없지만, 출력용 "월간 집계표"는 각 직원의 월 합계(우측 열)와 날짜별 합계(하단 행)가 있어야 원래 엑셀로
  관리하던 표와 같은 역할을 한다 — 그래서 `MonthGrid`를 재사용하지 않고 이 페이지에서 따로 표를 그렸다.

### 도메인 타입 / DB 스키마

- [lib/overtime/types.ts](../lib/overtime/types.ts) — `Employee`, `Project`, `WorkRecord`, `DailySummary` 정의. `WorkRecord`가 유일한 저장 단위이고, `DailySummary`는 그리드/모달 표시용 파생 데이터(비저장)임을 명시. 필드명은 DB 컬럼명과 1:1 (snake_case, camelCase 변환 레이어 없음) — 이 코드베이스의 기존 타입(`PerformingProject` 등)과 동일한 방식.
- [supabase/migration_overtime.sql](../supabase/migration_overtime.sql) — 실제 테이블 정의 (2단계에서 확정). 컬럼 설명은 [docs/database.md](./database.md) 참고.
- `start_time`/`end_time`은 Postgres `time` 타입이 아니라 **`text`**로 저장한다. 연장근무는 자정을 넘기는 경우(예: 21:00~24:00, 다음날로 이어지는 경우)가 흔한데 `time` 타입은 `24:00` 이상을 표현할 수 없기 때문 — 계산은 항상 앱 레이어(`lib/overtime/time.ts`, 6단계)에서 처리한다.
- `hours`는 파생 가능한 값이지만 컬럼으로 저장한다. `start_time`/`end_time` 텍스트를 조회마다 파싱하지 않고, 그리드·대시보드 집계를 `SUM(hours)` 한 줄로 끝내기 위함. 저장 시점(6단계 CRUD)에 한 번만 계산해서 넣는다.
- `overtime_employees.is_active` / `overtime_projects.status`로만 비활성화하고 행을 삭제하지 않는다. FK도 `ON DELETE RESTRICT`로 걸어, 과거 `overtime_work_records`가 남아있는 한 직원·프로젝트 행 자체가 삭제되지 않도록 막는다.
- RLS는 기존 테이블들과 동일하게 "인증된 사용자 전체 접근" 정책만 건다 — 실제 접근 제어(허용 사용자 목록)는 `app/(dashboard)/layout.tsx`에서 이미 처리되기 때문.
