# 휴가관리 — 계산·공휴일·검증 규칙

계산 로직은 전부 `lib/leave/calc.ts`의 순수 함수로 둔다(연장근무 `lib/overtime/time.ts`와
같은 위치·방식). 날짜는 `YYYY-MM-DD` 문자열로 다루고, 요일 계산이 필요할 때만
`new Date(y, m-1, d)` 로컬 생성자를 쓴다 — `new Date("YYYY-MM-DD")` 파싱 금지
(`docs/conventions.md` UTC 버그).

## 1. 전체 기간 (몇 박 며칠)

```
며칠 = 종료일 − 시작일 + 1
몇 박 = 며칠 − 1
```

- 같은 날짜: 0박 1일 / 8/1~8/3: 2박 3일 / 금~월: 3박 4일.
- 주말·공휴일 포함한 달력 기준. **차감일수와 별개 개념으로 항상 분리 표시·저장**
  (records에 `total_calendar_days` 저장, 박은 표시 시 −1).

## 2. 실제 차감일수 — 날짜별 전개 `expandLeaveDates()`

입력: 시작일, 종료일, 시작일 단위, 종료일 단위, 유형(차감 여부), 공휴일 Set.
출력: 날짜별 행 배열(03 문서의 `leave_record_dates` 구조).

날짜마다 순서대로 판정한다:

1. **토요일·일요일** → 차감 0, `is_weekend`.
2. **공휴일/회사휴무** (`holidays` 테이블에 있는 날짜) → 차감 0, `is_holiday` + 이름 스냅샷.
3. 평일이면 단위별 차감:
   - 시작일 → 시작일 단위 (전일 1 / 오전·오후 반차 0.5)
   - 종료일 → 종료일 단위
   - **중간 날짜 → 전일 1** (요구 8: 중간의 정상 근무일은 전일로 계산)
   - 시작일 = 종료일이면 단위 하나만 적용
4. **유형이 연차 미차감**(경조·병가·공가·대체휴무·기타)이면 위 결과와 무관하게
   차감 0으로 덮어쓴다 — 날짜 행은 그대로 남겨 "날짜 점유"는 기록한다(중복 검증용).

`실제 차감일수 = 날짜별 차감의 합`. 예: 금(전일)~월(전일) → 금 1 + 토 0 + 일 0 + 월 1 = 2일.
예: 8/10 오후 반차 0.5 + 8/11 전일 1 + 8/12 오전 반차 0.5 = 2일.

## 3. 집계 (조회 시점 계산, 저장 안 함)

연도 조회 시 그 해의 `leave_record_dates`(records join)를 한 번에 불러와 클라이언트에서:

- **월별 사용일수** = 직원별·월별 `deducted_days` 합 — 날짜가 속한 달 기준이므로
  월을 걸치는 휴가(7/31~8/1)는 자동으로 7월 1일·8월 1일로 나뉜다. 시작 월에
  몰아서 합산하는 일이 구조적으로 불가능하다.
- **연간 사용일수** = 1~12월 합.
- **잔여 연차 = granted_days + adjustment_days − 연간 사용일수.**
- 수정/삭제 시 dates를 재생성/삭제(CASCADE)하고 화면은 재조회 — 별도 "재계산 저장" 없음.

집계 월 기준은 **달력 월(1일~말일)**이다. 연장근무의 급여기간(전달 21~이번달 20)과
다르다 — 연차는 연 단위 정산이라 달력 월이 자연스럽고, 참고 엑셀도 1~12월 구조다.
(06 문서 결정사항 ①로 확인 요청)

## 4. 공휴일 처리

- **차감 계산은 오직 `holidays` DB 테이블만 본다** → 인터넷 없이 동작.
- 시드: 2026년 법정공휴일(대체공휴일 포함)을 마이그레이션에 넣는다.
- 갱신 경로 2가지:
  1. 공휴일 관리 모달 「{연도} 공휴일 인터넷에서 불러오기」 — 기존 `/api/holidays`
     (date.nager.at 프록시, 대시보드 캘린더가 이미 사용 중)에서 받아 upsert
     (`onConflict: holiday_date`). 실패(오프라인) 시 안내만 하고 기존 데이터 유지.
  2. 수동 추가/수정/삭제 — 회사 지정 휴무일(`holiday_type: 회사휴무`)도 같은 테이블.
- 저장된 휴가의 날짜 행에는 당시 공휴일 여부가 스냅샷으로 남는다. 공휴일을 나중에
  바꿔도 기존 휴가의 차감이 소급 변경되지 않는다 — 소급이 필요하면 그 휴가를 열어
  다시 저장(재전개)하면 된다. 매년 초 다음 해 공휴일을 불러오는 것을 운영 절차로 문서화.

## 5. 저장 검증 (클라이언트, 저장 전 순차 검사)

| # | 검사 | 처리 |
|---|---|---|
| 1 | 필수값 누락 (직원·유형·시작일·종료일) | 차단 |
| 2 | 종료일 < 시작일 | 차단 |
| 3 | 입사일(hire_date) 이전 날짜 포함 | 차단 |
| 4 | 퇴사일(resign_date) 이후 날짜 포함 | 차단 |
| 5 | 같은 직원의 기존 휴가와 날짜 겹침 | 아래 단위 규칙으로 판정, 겹치면 차단 |
| 6 | 차감일수 0인 전일반 기간 (전부 주말/공휴일) | 경고만 (기록 자체는 허용 — 유형에 따라 정상) |
| 7 | 잔여 연차 초과 (차감 유형인 경우) | **경고 + confirm 후 저장 허용** |

겹침 판정(같은 날짜의 기존 `leave_record_dates` 행과 새 행 비교, 수정 시 자기 record 제외):

- `full` vs 무엇이든 → 겹침
- `am` vs `am` → 겹침 / `pm` vs `pm` → 겹침
- `am` vs `pm` → 허용 (오전 반차 + 오후 반차)

잔여 초과는 MVP에서 "경고 후 허용"이며, 차단으로 바꿀 수 있도록 검증 함수가
`{ level: 'block' | 'warn' }`을 반환하는 구조로 만든다 — 추후 설정 테이블이 생기면
이 레벨만 바꾸면 된다.

## 6. 연차 설정 검증

부여/조정 저장 시: `새 최종(부여+조정) < 그 연도 기 사용일수`이면
"이미 사용한 연차일수(N일)보다 부여 연차일수가 적습니다" 경고 표시(저장은 허용 —
정정 입력 중일 수 있으므로 차단하지 않고, 메인 테이블 잔여가 음수·위험색으로 드러난다).
