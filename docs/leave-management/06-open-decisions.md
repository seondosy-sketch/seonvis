# 휴가관리 — 결정 필요 사항

구현 전 사용자 확인이 필요한 항목. 결정되면 이 문서에 결과를 기록한다.

## ① 월 집계 기준 — 달력 월 vs 급여기간

연장근무는 "전달 21일~이번달 20일" 급여기간을 쓰지만, 휴가관리는 **달력 월(1일~말일)**로
설계했다(참고 엑셀이 1~12월 구조 + 연차는 연 단위 정산). 급여기간 기준이 필요하면
집계 함수만 바꾸면 되는 구조지만, 테이블 열 의미가 달라지므로 먼저 확정 필요.

**제안: 달력 월.** → 결정: 제안대로 승인 (2026-07-14)

## ② 입사일/퇴사일 편집 위치

연장근무 직원 관리 화면은 수정 금지라, `overtime_employees`의 신규 컬럼(hire_date,
resign_date) 편집을 **휴가관리의 연차 설정 모달**에 뒀다. 직원 등록/삭제 자체는 여전히
연장근무 직원 관리에서 한다(휴가관리는 편집만).

**제안: 연차 설정 모달에서 편집.** → 결정: 제안대로 승인 (2026-07-14)

## ③ 공휴일 시드 범위

마이그레이션에 2026년 법정공휴일만 시드하고, 2027년 이후는 연초에 「인터넷에서
불러오기」 버튼(기존 `/api/holidays` 재사용) 또는 수동 입력. 2027년까지 미리 시드할 수도
있다(공표된 대체공휴일은 연말에야 확정되는 경우가 있어 어차피 보정 필요).

**제안: 2026년만 시드 + 불러오기 버튼.** → 결정: 제안대로 승인 (2026-07-14)

## ④ 잔여 연차 초과 저장

요구대로 MVP는 "경고 + 확인 시 저장 허용". 검증 함수가 차단/경고 레벨을 반환하는
구조로 만들어 추후 차단 모드 전환이 쉽게 되어 있다. 별도 설정 화면은 MVP에서 안 만든다.

**제안: 경고 후 허용(요구사항 그대로), 설정 화면은 추후.** → 결정: 제안대로 승인 (2026-07-14)

## ⑤ 공휴일 소급 반영

이미 저장된 휴가는 저장 당시 공휴일 기준의 차감이 스냅샷으로 유지된다. 공휴일을 나중에
수정해도 자동 소급하지 않고, 필요하면 해당 휴가를 열어 다시 저장하면 재계산된다.
(자동 소급은 "어떤 휴가가 영향받는지" 추적 복잡도가 커서 MVP 제외.)

**제안: 소급 없음 + 재저장으로 수동 반영.** → 결정: 제안대로 승인 (2026-07-14)

## ⑥ 삭제 방식

soft delete(deleted_at) 대신 MVP는 일반 삭제(확인창 + CASCADE로 날짜 행 정리).
요구 15에서 "복잡도가 높다면 MVP는 일반 삭제로 시작" 허용됨. 연차 부여/조정 이력은
별도 테이블로 남는다(휴가 기록 삭제 이력과는 다름).

**제안: 일반 삭제.** → 결정: 제안대로 승인 (2026-07-14)

## ⑦ 휴가 유형 관리 UI 포함 여부

기본 8종 시드는 확정. 유형 추가/수정(차감 여부·단위·순서·활성화) 관리 모달을 MVP에
포함하는 것으로 계획했다 — 요구 5에 "설정할 수 있도록 설계"가 명시돼 있어서.
단순화를 원하면 시드 고정으로 시작하고 관리 UI는 다음 단계로 뺄 수 있다.

**제안: MVP 포함(단순한 관리 모달).** → 결정: 제안대로 승인 (2026-07-14)

## ⑧ 반차 유형과 사용 단위의 관계

"오전/오후 반차"가 휴가 유형에도, 날짜별 사용 단위에도 존재한다. 설계상 **유형은
분류·차감여부용, 실제 차감 계산은 날짜별 단위 기준**이다. 모달에서 유형 "오전 반차"
선택 시 같은 날 + 오전 단위로 자동 세팅해 이중 입력 혼란을 줄인다. 여러 날에 걸친
반차 조합(시작일 오후 + 종료일 오전)은 유형 "연차"로 입력하는 것을 기본 안내로 한다.

**제안: 위 방식.** → 결정: 제안대로 승인 (2026-07-14)
