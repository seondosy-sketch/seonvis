# 현장 현황 — 결정 필요 사항 (2026-07-15 사용자 확정 반영)

번호·감리원·시공사·날짜 없이 주소만 남은 행. "LH 천안신부 행복주택(감독권한)"(#6, 정상

## ③ "LH 천안 역세권시도시 재생사업" 중복(실은 빈 껍데기 행 2건) — 확정: 제외 + 확인 필요

01 문서 재검증 결과: 이 이름의 행은 **주택법 #8, 건축법 #8** 두 곳에 있고, **둘 다
감리원·전화·시공사·주소·날짜·비고가 전부 완전히 공란**이다("주요 정보가 동일 = 둘 다
아무 정보도 없음"에 해당). 건진법 #7의 정상 데이터 현장("역세권신도시", 정상 철자,
감리원/시공사/기간 전부 존재)과는 시트도 다르고 완성도도 달라 그 항목의 중복도 아니다.
**두 행 모두 실질 데이터가 없어 "별도 계약"인지조차 판단할 근거가 없으므로 이관하지 않고
확인 필요 목록에 기록** — 실제 현장이면 화면에서 정보를 채워 새로 추가.

## ④ 비활성 방식 — 확정: active만 사용

`deleted_at` 생략, `active` boolean만. 완전 삭제 UI는 MVP에서 제공하지 않는다(사용자
결정 — 상세 모달은 [비활성화]/[재개]만 제공, 완전 삭제는 확장으로 미룸).

## ⑤ 전통소 표시명 — 확정: "분리발주(전기·통신·소방)"

내부 `source_category`는 원본 "전통소" 그대로 보존(향후 동기화 매칭용). 화면/필터에 쓰는
표준값 `legal_category`는 "분리발주(전기·통신·소방)"로 저장·표시(원본값과 표준 표시값을
컬럼으로 분리 — 04 문서).

## ⑥ 연락처 처리 — 확정: 원본 보존 + 다중 추출 필드 + 불확실 표시

- `site_phone_raw`: E열 원본 텍스트 그대로(정보 손실 없음)
- `site_landline`: 추출된 현장 유선전화(추정, 여러 개면 세미콜론으로 이어붙임 — 버리지 않음)
- `manager_mobile`: 괄호 안 첫 010 번호(추정 — "확정된 책임자 번호"라고 단정하지 않음)
- `phone_uncertain` boolean: 번호가 여럿이거나 패턴이 모호하면 true — 상세 화면에
  "확인 필요" 표시, 원본(`site_phone_raw`)은 항상 함께 보여준다.

## ⑦ 준공 임박 기준 — 확정: 90일 이내(당일 포함)

준공예정일까지 0~90일 = 준공 임박, 91일 이상 = 진행 중, 준공예정일 **당일**은 준공 임박에
포함, 다음 날부터 준공 완료.

## ⑧ 진행 상태 — 확정: 자동 6종 상태 + 수동 오버라이드(자동 포함 6개 선택지)

선택지: 자동 / 착수 전 / 진행 중 / 준공 임박 / 준공 완료 / 중지. "자동"을 고르면
`manual_status = null`로 저장돼 날짜 기준 자동 계산, 나머지는 수동 고정 + 뱃지에 "· 수동"
표식.

## ⑨ site_code와 향후 동기화 — 확정: 우선순위 조정

1. 엑셀에 `site_code`가 있으면 그 값으로 매칭(향후 내보내기 파일 재이입 시)
2. 없으면 정규화 현장명 + `legal_category`(원본 `source_category` 사용) + 주소 조합
3. 시공사·공사기간을 보조 비교값으로
4. 불확실하면 사용자 확인 대상 (05 문서에 반영)

## ⑩ 개인정보 시드 — 확정: 저장소에 커밋하지 않음

전화·주소가 포함된 이관 데이터는 SQL 파일로 커밋하지 않는다. 마이그레이션(스키마)만
`supabase/migration_sites.sql`로 커밋하고, 89건 실데이터는 **구현 시점 1회성 로컬
스크립트가 엑셀을 직접 읽어 Supabase REST로 삽입** — 스크립트·산출물은 세션
스크래치패드에만 남기고 리포지토리·`.gitignore`(`/supabase/seed_*.sql`)로 재차 방지.
보고는 건수만 하고 전체 개인정보는 로그·문서에 남기지 않는다(기술인 주소록과 동일하게
소급 정리함 — engineers도 이번에 같이 정리).
